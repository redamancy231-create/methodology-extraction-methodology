## 详细分析

### 1. 分析口径与证据边界

本证据卡只依据提示词中内联的源文件内容，不使用外部检索，也不参考或推测其他模型的提取结论。评估对象以 **baopinshui 的原始 NPGS 项目**为主；fork 作者后来添加的 `CLAUDE.md`、`AGENTS.md`、`CONTRIBUTING_UPSTREAM.md`、`NPGS_ANALYSIS.*` 和 `_reviews/` 只用于说明 fork 层发生了什么，不能反向归因给原作者。该边界由提示词第 26—33 行明确规定。

状态含义：

- **✅**：存在直接、较完整且可核查的内联证据。
- **⚠️**：存在部分痕迹，但关键环节、理由或来源不完整。
- **❌**：内联材料明确表示不存在，或没有足以支持该组件存在的证据。

“可提取组件数”统计 ✅ 与 ⚠️，因为两者都能提取到非空特征；❌ 表示该组件在现有证据中不可提取。

- 原作者 README 再次给出项目名称，并在“开源协议”中声明“GPL-3.0。原作者：baopinshui”（第 102—110、160—161 行）。
- Git 历史概览给出约 `2025-02 ~ 2026-07` 的原始提交跨度，以及 `2026-05 ~ 2026-07` 的活跃开发期（第 236—243 行）。

**判断**

C1 所要求的五类信息均有直接证据。开发周期是概览级区间而非精确首末提交日期，但已足以支持“可追溯”的判断。

**归属说明**

上述身份和历史信息归属于原作者项目；fork 地址只是项目关系信息，不代表 fork 作者是 NPGS 的源作者。

### C2 架构/组织可提取：✅

**发现**

- README 将项目划分为 `NPGS/Sources/Program/`、`NPGS/Sources/Engine/Core/`、`NPGS/Assets/` 和 `NPGS/Tools/`，并逐项说明游戏逻辑、引擎框架、资产与工具职责（第 154—158 行）。
- 功能章节进一步揭示黑洞渲染、星系生成、文明模拟、图形/UI 等主要能力域（第 124—146 行）。
- 技术栈与构建段落记录 C++20、Vulkan、GLFW、ImGui、VS2022、vcpkg 等实现环境（第 148—152 行）。

**判断**

可以从原作者 README 提取出清晰的顶层模块图、职责分配和主要技术边界，因此评为 ✅。这不等同于拥有完整架构设计文档：例如内部依赖图、接口契约和运行时部署关系仍不完整，但不影响“架构/组织可提取”的成立。

**归属说明**

### C3 设计意图可提取：⚠️

**发现**

**判断**

**判断**

**可提取内容**

- 可以定位数据承载位置：`NPGS/Assets/` 下的 CSV 恒星数据表。
- 可以识别部分模型或方法名称：IMF、恒星演化链、Kerr 度规、RK4、Kerr-Schild 坐标等。

**缺口**

内联材料没有给出 CSV 数据的原始数据集、作者/机构、版本、下载地址、许可证、生成过程、字段说明、校验和或更新时间；也没有为物理模型提供论文、教材、公式来源、参数依据及数值验证记录。因而可以追踪到“仓库中的数据文件和算法名称”，但不能继续追踪到外部来源及结论验证。

**判断**

评为 ⚠️，而不是 ✅：仓库内位置可追踪，但科学数据和模型依据的来源链不完整。

## 综合结论

NPGS 的强项是 **项目身份、功能范围和代码组织清晰可提取**：从原作者 README 可以较可靠地重建它是一套以 C++/Vulkan 实现、横跨天体物理模拟、图形渲染和文明模拟的手写系统。Git 概览也保留了最低限度的开发历史痕迹。

其方法论证据的主要短板是：宏观愿景多、显式决策理由少；能看到提交历史，但看不到制度化开发流程；原项目没有 AI 协作文档，也没有可验证的独立审查、测试/CI 或修复验证闭环；数据文件和模型名称可以定位，但科学来源链未被记录。

因此，本次独立提取将其定为 **证据等级 C、5/8 个组件存在可提取痕迹、提取层级为“部分提取”**。其中只有 C1 与 C2 达到完整支持；C3、C4、C8 为部分支持；C5、C6、C7 在原项目范围内无可验证证据。Fork 作者后来建立的 AI 边界、影响分析和审查计划体现了 fork 层的方法论增强，但不能用于抬高原作者项目的原生方法论评级。
