## 一、总判定

claude-skills 可提取为一套覆盖项目生命周期关键节点的 AI 协作治理方法，而不只是三个互不相关的提示词或模板：

1. 建设前由 kill-test-first 判断“值不值得做”，用事前否决、死亡判据、强基线和预登记抑制无效建设及事后改口径。
2. 项目初始化或治理重构时由 write-claude-md 把真正会改变 Agent 行为的规则压缩进项目级 CLAUDE.md。
3. 会话结束时由 session-end 将状态、记忆、教训和下一步结构化交接给后续会话。
4. 三者外层再由路由器模式、渐进式披露、工具白名单、只读审查证据、同步链和停止条件提供治理约束。

核心方法论可概括为：

**先否决错误方向，再最小化并显式化 Agent 约束，最后把跨会话状态闭环交接；全过程以可停止、可审计、可追溯和生命周期感知为设计目标。**

“8/8 可提取”表示八个组件均能从所给材料中找到相关信息，并不表示八项均已充分验证。当前 C2 与 C5 有较强直接文档证据；其余组件存在时间线、原始审查件、实际变更记录或逐项溯源材料缺失，因此总体为“主要提取”，不是“完整提取”。

## 二、证据等级说明

**总体证据等级：B。**

- README.md 与 CLAUDE.md 是项目自身的一手治理文档，对架构、意图、Agent 边界、同步规则和更新协议具有直接证明力。
- 两份文件存在交叉印证：README 描述三个 Skill 的职责与组合关系，CLAUDE.md 将设计转化为维护约束。
- 提示词未提供实际 SKILL.md、CHANGELOG.md、project_status.md、verified_by frontmatter、reference/death-criteria.md 或原始审查记录。
- “50+ 轮、5 种后端、每个 Skill 至少 2 个异后端审查、发布前经过 kill-test”等只能判定为项目自述，不能在本次材料范围内升级为 A 级已核验证据。

## 三、C1—C8 详细分析

### C1 项目基础信息：⚠️

**已提取**

- 名称：claude-skills／Claude Skills。
- 作者及维护者：redamancy231-create。
- 仓库：GitHub 上的 redamancy231-create/claude-skills。
- 类型：非软件项目，Markdown 文档型 Claude Code Skill 集合。
- License：CC BY 4.0。
- 当前状态：v1.0.1，CLOSED，脱敏闭合。
- 分发形态：三个可独立安装的用户级 Skill。

**方法论特征**

项目把生命周期状态作为治理输入，而非仅把版本号作为发布标签。CLOSED 状态改变文档权重和维护行为：强调历史坑位、重启条件与只读证据，弱化活跃开发命令。这说明生命周期元数据直接参与 Agent 决策。

**缺口**

未提供项目开始日期、里程碑日期、v1.0.0 到 v1.0.1 的发布时间或完整开发周期；也未提供 CHANGELOG.md 内容。README 尾部生成日期不能替代项目开发周期。因此判为 ⚠️。

### C2 架构/组织：✅

**直接证据**

README 给出三项共同设计原则：

- 路由器模式：SKILL.md 只保留核心流程和 Quick Reference 路由表。
- 渐进式披露：详细步骤、模板和脚本按上下文从 reference 等目录加载。
- allowed-tools 白名单：限制 Skill 上下文中的可用工具。

CLAUDE.md 将其固化为架构不变量：

1. SKILL.md 不超过 100 行。
2. 详细内容外置到 reference 目录并由路由表索引。
3. frontmatter 必须包含 name、description、allowed-tools、verified_by。
4. 每个 Skill 必须有 reference/death-criteria.md。

README 还给出三个 Skill 的生命周期关系：kill-test-first 位于建设前，write-claude-md 位于项目初始化或治理阶段，session-end 位于会话边界。

**可迁移方法论**

- 薄入口、厚参考件：把高频入口与低频知识分层，控制常驻上下文成本。
- 路由而非堆叠：主文件负责决定何时读取哪个参考件，不承载全部知识。
- 架构不变量显式化：行数上限、必填元数据和死亡判据均可机械检查。
- 生命周期编排：三个 Skill 按“决策—建档—交接”串联，而非平铺为工具箱。
- 能力最小化：allowed-tools 将最小权限原则应用到 Agent Skill。

架构、目录职责、路由机制、组合关系及不变量均有直接证据，判为 ✅。

### C3 设计意图：⚠️

README 对“为什么是这三个 Skill”给出清晰回答：它们来自真实项目工作流，分别处理三个高代价断点：

- 未经否决就启动错误方向。
- 项目文档不能有效约束 Agent。
- 跨会话上下文与状态丢失。

由此可提取五项设计意图：

1. 围绕失败模式选工具，而非围绕功能列表选工具。
2. 以行为后果检验文档价值：删除规则后 Agent 是否更容易犯错。
3. 把否决权前移：在投入前终止错误方向。
4. 把会话结束视为项目状态迁移事件，而非聊天结束。
5. 对活跃、维护和 CLOSED 项目采用不同治理权重。

**关键缺口**：提取框架要求识别“P0→P1→P2 重构协议”，但所给 README.md 与 CLAUDE.md 中没有 P0、P1、P2 的定义、迁移规则或实例。不能把步骤 0→8、双门协议或版本号规则擅自解释为该协议。因此判为 ⚠️。

### C4 开发流程：⚠️

CLAUDE.md 提供了规范性维护流程：

- v1.0.x 用于脱敏、措辞和同步修正；v1.x.0 用于架构变化或新增 Skill。
- 修改 SKILL.md 后检查根 README、英文 README、繁体 README、CHANGELOG.md 和 project_status.md。
- 用户可见变更需更新 CHANGELOG.md。
- 修改后运行三语术语同步、行数和 frontmatter 完整性检查。
- 新增审查记录写入 _review，历史审查证据只读。
- CLOSED 项目重启前先读历史教训和反模式；新 Skill 先过 kill-test-first，发布前再做异后端审查。

可归纳为：

**变更分类 → 权限判断 → 增量修改 → 多文件同步 → 机械检查 → 状态更新 → 必要时审查归档。**

三语 README 采用主从关系：简体中文根 README 是单一事实源，英文与繁体为派生版本，禁止在派生版本独立创造新内容。

但材料只展示“应如何更新”，没有 CHANGELOG 的真实条目、提交历史、版本标签或 project_status 内容，无法确认实际演进是否遵守协议。因此判为 ⚠️。

### C5 AI 协作方法论文档：✅

CLAUDE.md 覆盖了高价值 Agent 治理要素：

1. 项目类型和生命周期。
2. 无需确认、需先确认、禁止三层权限边界。
3. SKILL.md 修改后的同步链。
4. 多语言文档主从关系。
5. Skill 架构不变量。
6. 可执行检查命令。
7. 已知坑位、后果与预防措施。
8. 明确停止条件。
9. 版本规则、修改后必做事项和 CLOSED 项目重启指南。

其重点不是罗列所有文件，而是记录会改变 Agent 行为的边界、同步义务、不可破坏的不变量和高复发错误。这与 write-claude-md 的脆弱性测试和五滥检查一致。

**突出特征**

- 许可分层比模糊的“谨慎修改”更可执行。
- Agent 不只知道如何继续，也知道何时必须停止。
- 修改一个 Skill 后需要沿同步链检查派生文档和状态文件。
- write-claude-md 自身变更会反过来影响本项目 CLAUDE.md，文档识别了这种自指风险。
- 规则配套 grep、wc 和 frontmatter 检查命令。
- _review 只读，避免为迎合当前结果而改写历史证据。

就文档设计完整性而言，C5 判为 ✅；这不代表所有命令和 Skill 行为已在本环境实测。

### C6 独立审查/质量保障：⚠️

README 声称框架经历 50+ 轮、跨 5 种 LLM 后端验证；每个 Skill 至少由 2 个异后端独立审查；审查者零卷入设计；每个 Skill 发布前经过自身 kill-test。CLAUDE.md 又规定 verified_by 为受保护的必填元数据，_review 历史记录只读，新 Skill 发布前须做至少 2 个异后端审查，并禁止在无运行环境时声称已验证。

可提取的方法论包括：

1. 设计者与审查者分离，降低自我确认偏差。
2. 使用异后端而不是只做同模型重复采样。
3. 事前 kill-test 与发布前独立审查分别控制方向错误和产物缺陷。
4. 验证声明属于受保护元数据，不能作为普通文案随意修改。
5. 严格区分静态检查与行为验证。

但本次未提供 verified_by 实值、_review 记录、审查 prompt、输出、冲突解决记录、kill-test 实例或 50+ 轮清单。因此只能确认质量保障制度被写入文档，不能独立确认所述审查已经发生，判为 ⚠️。

### C7 闭环修复：⚠️

材料呈现了一个规范性的修复闭环：

1. 审查或维护发现问题。
2. 按权限边界进行增量修复，禁止全量重写。
3. 沿同步链更新三语 README、CHANGELOG 和 project_status。
4. 用术语 grep、行数和 frontmatter 检查验证一致性。
5. 如有新增审查，将记录归档到 _review。
6. session-end 记录状态变化、教训和 Next Steps，供后续会话继续。

停止条件为闭环增加失败出口：变更会破坏三语关系、超过行数上限、削弱死亡判据或绕过主版本时，应停止而不是强行完成。

这体现了“修复必须带同步验证”“增量修复优先于重写”“状态交接属于闭环”“停止也是合法结论”。但没有具体的“审查发现 → 补丁 → 复审结果”案例，也没有 diff 或 review resolution，因此判为 ⚠️。

### C8 数据溯源：⚠️

README 给出方法论上游 AI 协作框架 §9.2–§9.3、审查方法来源 Independent Review Toolkit、同系实验协议 Prompt-TDD Methodology，并声明每个 Skill 的 frontmatter 应记录 provenance 和 verified_by；CLAUDE.md 将 _review 定义为只读历史证据。

可提取的方法论包括：

- 逐 Skill 溯源优于仅项目级致谢。
- 来源、验证声明和原始审查记录分层存放。
- 历史证据只读以减少事后篡改。
- README 区分方法论上游、审查来源、同系项目和实际使用关系。

但实际 SKILL.md frontmatter 与 _review 记录未提供；从某条主张到上游段落、审查轮次、修复提交的逐项映射也不存在。尤其 _review 不提交仓库会削弱公开复核能力。因此项目有较强“溯源架构”，但当前证据包不足以完成“溯源验证”，判为 ⚠️。

## 四、跨组件方法论模型

### 1. 三阶段生命周期控制面

| 阶段 | Skill | 主要风险 | 控制方式 | 产物 |
|---|---|---|---|---|
| 建设前 | kill-test-first | 在错误方向上投入 | 可证伪命题、先验对标、最强反方、死亡判据、最便宜证伪、强基线 | GO／STOP／REDESIGN；GO 时形成预登记 |
| 初始化或治理 | write-claude-md | Agent 缺少有效约束，或文档充满噪声 | 脆弱性测试、五缺检查、五滥检查、实证规则、多后端验证 | 行为导向的项目级 CLAUDE.md |
| 会话边界 | session-end | 状态、记忆和教训丢失 | 状态更新、记忆交叉检查、教训捕获、Retrospect 候选、标准化 Next Steps | 可继续执行的交接状态 |

三个阶段形成控制循环：kill-test-first 控制是否进入，write-claude-md 控制 Agent 如何行动，session-end 控制行动状态如何连续。

### 2. 双层质量门

- **方向门**：建设前用死亡判据和便宜证伪阻止不值得做的项目。
- **发布门**：产物完成后用异后端独立审查和同步检查阻止未经验证或不一致的发布。

方向门降低机会成本，发布门降低缺陷外溢。两者不能互相替代：通过 kill-test 只说明值得尝试，不说明实现正确；通过发布审查也不能追回已经浪费的建设成本。

### 3. 文档即控制系统

项目中的 Markdown 不只是说明书，而是控制面：

- frontmatter 控制工具能力和验证元数据。
- CLAUDE.md 控制修改权限、停止条件和同步义务。
- project_status.md 控制生命周期状态。
- memory 文件控制跨任务经验保留。
- CHANGELOG 控制用户可见变更历史。
- _review 控制审查证据归档。

这是一种“可执行治理文档”方法：即使规则不是编译器强制，也尽量配套检查命令、目录不变量和停止条件。

### 4. 单一事实源加派生同步

三语 README 采用根 README 为主版本、英文和繁体为派生版本的模式。它牺牲多语言版本的独立编辑自由，换取一致性和较低维护成本。同步风险通过主从关系、禁止派生版本独立创作、修改后 grep 关键术语，以及把遗漏同步列为坑位和停止条件来控制。

### 5. 证据诚实边界

“本机没有 Claude Code 运行环境时不得声称已验证或已测试”是重要治理规则。它将文档检查、静态一致性检查和行为验证分开，避免把弱证据包装成强结论。本证据卡沿用该原则：项目关于多后端审查的陈述被记录为自述，而非独立核验事实。

## 五、主要优势

1. 覆盖方向选择、Agent 治理和会话连续性三个高代价节点。
2. 规则行为化，能够回答 Agent 应做什么、何时停、改后检查什么。
3. 路由器模式为上下文成本提供明确结构约束。
4. 前置死亡判据降低事后合理化风险。
5. 治理规则能根据 ACTIVE、维护、CLOSED 等生命周期调整。
6. 多语言漂移、自指文档和验证声明修改等风险被显式建模。
7. 增量修改、历史证据只读和验证声明受保护，有利于审计可比性。

## 六、主要风险与不可验证项

1. 审查声明可见，但原始审查证据不可见。
2. 缺少实际版本历史，规范协议不能证明历史上始终遵守。
3. 缺少实例级“缺陷—修复—复审”闭环。
4. P0→P1→P2 协议在源材料中缺失。
5. verified_by 未展示，字段语义和粒度不可核验。
6. 50+ 轮的统计口径未定义。
7. “实战验证”与“静态审查”可能被混用；没有运行日志时不能扩大解释为运行时有效性验证。
8. 脱敏闭合与公开可追溯性存在张力：公开材料越少，第三方越难核验审查链。

## 七、升级为 A 级或完整提取的最小补证清单

1. 提供三个实际 SKILL.md，核验行数、路由表、allowed-tools、verified_by 和 provenance。
2. 提供 CHANGELOG.md 与版本标签时间，恢复 v1.0.0→v1.0.1 的实际演进。
3. 每个 Skill 至少提供一份脱敏审查记录，包含后端、日期、输入版本、发现、处置和复审结论。
4. 提供一条完整“发现→修复 diff→验证→关闭”的闭环证据。
5. 提供 kill-test 预登记样例，证明死亡判据在查看结果前冻结。
6. 定义 50+ 轮的计数口径并给出轮次索引。
7. 提供 P0、P1、P2 的正式定义、迁移条件和至少一个重构实例；若项目没有该协议，应从提取框架中删除该预期。
8. 为 README 中的上游方法论关系增加逐条映射，而不是只提供项目级链接。

## 八、最终结论

claude-skills 的可迁移价值主要不在三个 Skill 的具体措辞，而在其组合出的治理结构：

- kill-test-first 把“是否值得做”变成建设前的正式决策门。
- write-claude-md 把项目知识压缩成真正改变 Agent 行为的最小约束集。
- session-end 把每次会话的状态变化转化为下一会话可消费的交接物。
- 路由器模式、最小权限、同步链、停止条件、只读审查证据和生命周期状态约束整个系统。

基于本次提供的 README.md 与 CLAUDE.md，可以对全部 C1—C8 做有边界的提取，但只能确认方法论和治理协议被清楚设计并记录。对于多后端审查是否实际完成、闭环修复是否真实发生、逐 Skill 溯源是否完整，以及 P0→P1→P2 是否存在，现有材料不足以独立验证。

**最终评级：证据等级 B；8/8 组件可提取；提取层级为主要提取；不满足完整提取。**
