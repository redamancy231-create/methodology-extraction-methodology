### C2 架构与组织 — ✅

**可提取结论**

项目采用分层的 Python/C++ 混合架构：

1. **归档行为基线**：V3.3.py 被定义为不可重跑、不可继续修改的上游基线。
2. **Python 纯计算参考层**：src/core 下拆出 6 个模块，覆盖 DTW、形态匹配、技术指标、市场特征、风控和统计指标，并移除掘金 SDK 依赖。
3. **C++ 加速层**：src/cpp/etf_core.cpp 通过 pybind11 暴露计算接口，使用 C++20 和 MSVC Release /O2。
4. **桥接契约层**：以 py::array_t 的 dtype/连续布局约束、固定返回结构、异常校验、GIL 生命周期和类型存根约束 Python/C++ 边界。
5. **验证层**：单元测试验证 Python 行为，verify_etf_core.py 和 verify_batch.py 负责 C++ 与 Python 的一致性及批量接口验证。
6. **展示层**：Jupyter notebook 用于交互演示，并接受独立的门面审查。

**关键架构方法**

- 先把平台绑定代码与纯计算逻辑分开，再进行跨语言迁移。
- Python 实现不是被立即删除，而是保留为 C++ 实现的行为 oracle。
- 对耗时纯 C++ 段释放 GIL，在构造 Python 返回对象前重新持有 GIL。
- 单次接口与批量接口分离，避免把算法加速和跨语言调用开销混为一谈。

**内部口径问题**

- README 项目树称统一 C++ 模块有 **8 个函数**，CLAUDE.md 项目树称有 **7 个函数**。这不影响架构存在性，但说明文档同步尚未完全闭合。

## 三、独立提取出的核心方法论模式

### M1 封存基线与非目标声明

先把原始 V3.3 定义为不可继续演化的行为基线，同时明确新仓库不是交易系统、不是收益改进项目。这样可防止重构过程中把工程优化、算法修改和策略绩效混在一起。

### M2 Functional Core 提取

从平台绑定的长脚本中提取无 SDK 依赖的纯计算模块。纯函数化是后续单测、差分验证、C++ 替换和独立 benchmark 的前置条件。

### M3 双实现语义锁定

保留 Python 参考实现，以它作为 C++ 版本的行为 oracle；对关键输出设置显式浮点容差，而不是只判断程序是否运行成功。这是一种“替换实现、不替换语义”的迁移策略。

### M4 桥接层契约优先

把 py::arg 数量、dtype 强制转换、连续内存、索引类型、返回键顺序、异常类型、空结果和 FEATURE_KEYS 纳入正式契约。混合语言工程的主要风险往往不在核心算法，而在语言边界。

### M5 GIL 分段管理

只在纯 C++ 长计算区释放 GIL，在任何 Python 对象构造和返回前恢复持有状态；对 early return 使用可推理的结构，避免把 GIL 生命周期散落在多个返回分支中。

### M6 分层性能声明

至少区分三种指标：

1. 核心算法 Python 对 C++；
2. 单次端到端形态匹配；
3. C++ 单次调用多次对 C++ 批量调用一次。

并用 Amdahl 定律解释局部加速与整体吞吐的差异，避免把不同分母的 speedup 放在同一叙事中。

### M7 异后端角色化审查

不是让多个模型重复做同一种“看一遍代码”，而是分配魔鬼代言人、完备性、代码改进、回归和门面准确性等角色，使审查维度互补。

### M8 发现—修复—测试闭环

高价值发现应落为代码契约、参数守卫、测试或文档修订。例如 match_step 守卫和 52→54 测试增长，说明至少部分审查发现已进入可回归资产。

### M9 项目内代理治理

把影响分析、风险提示、变更检测、禁止普通字符串重命名等规则写入仓库文档，让后续代理在新会话中仍受同一操作协议约束。

### M10 可复现交付面

README 给快速构建入口，验证脚本承担语义复核，benchmarks 承担性能复核，notebook 承担演示，CLAUDE.md 承担经验和治理沉淀。不同产物各有职责，避免单一 README 承载全部证据。
