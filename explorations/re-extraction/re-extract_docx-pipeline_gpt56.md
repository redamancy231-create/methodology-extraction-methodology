## 1. 提取范围与判定口径

本次仅依据提示词中内嵌的两份一手项目文档进行独立提取：README.md 与 CLAUDE.md。未引用或对照 DeepSeek 的既有证据卡，也未把提示词中的项目概述直接当作已核验事实；概述仅用于发现材料间可能存在的口径冲突。

证据等级判定：

- **A**：有直接产物、日志、提交、测试报告或可复核记录，并且来源间一致。
- **B**：有一手项目文档、具体路径、命令和结构信息，但部分关键产物未随材料提供。
- **C**：主要是概述性或自述性陈述，缺乏可定位的实现或过程证据。
- **D**：缺少证据，或主要结论存在无法解释的冲突。

本项目总体定为 **B**：两份一手文档提供了较完整的架构、治理、审查与修复线索，但未提供 CI 配置、CHANGELOG 正文、提交历史、原始审查报告和测试清单；测试数、审查轮数及 GitNexus 指标还存在版本漂移。

## 2. 总体结论

从材料中可以稳定提取出一套 面向高风险文档生成工具的工程化方法论：

1. **以用户痛点定义边界**：聚焦 Markdown → 中文 DOCX，不把反向提取混入产品职责。
2. **以双后端隔离能力与依赖风险**：Pure Python 提供可控排版和低依赖兜底，Pandoc 提供成熟解析、代码高亮和更高质量数学公式。
3. **把转换过程组织成显式管道**：预处理 Mermaid、选择后端、生成文档、统一后处理、原子保存与备份。
4. **把配置当作公共契约**：dataclass、YAML 模板和 JSON Schema 三者共同约束配置演进。
5. **把设计不变量写入 Agent 文档**：明确可修改范围、禁止事项、影响分析和提交前变更检测。
6. **把质量保障做成多层闭环**：自动化测试、跨平台 CI、模型审查、人工目视验证和已知限制共同工作。
7. **允许修复不了但可以被治理的结论**：对 Pure Python 数学渲染等缺陷不虚构闭合，而是降级、标注实验性并推荐 Pandoc。
8. **保留过程来源，但尚未达到强审计链**：文档记录模型、日期、轮次、发现与内部路径，却缺少提交哈希、测试报告和发布产物校验值。

因此，该项目的方法论特征不是单纯的写一个转换器，而是：**用架构分流、契约约束、独立审查、人工验收和可降级策略，控制异构文档生成链路中的质量风险。**

## 4. C2 架构与组织证据卡

**判定：✅ 架构边界、模块结构和主要执行阶段均清晰。**

### 架构证据

README 给出的主流程是：

**Markdown 输入 → Mermaid 检测与 PNG 预渲染 → 后端选择 → Pure Python 或 Pandoc 转换 → 字体、表格、TOC 等后处理 → DOCX 输出。**

项目结构进一步把职责拆为：

- **cli.py**：init、convert、validate、info 四类入口。
- **config/**：默认值、YAML 加载、dataclass 配置模型、验证器。
- **converters/**：抽象转换器、Markdown 状态机解析器、Pure Python 后端、Pandoc 后端、共享工具。
- **renderers/**：Mermaid 预渲染。
- **data/**：JSON Schema 与四套 YAML 模板。
- **utils/**：编码与路径规范化。
- **tests/**：基础测试和 fixture。

CLAUDE.md 补充了两条关键结构约束：PandocConverter 与 PurePythonConverter 共同继承 AbstractConverter；project.yaml 经 load_config 转为 DocxPipelineConfig，再进入 validate_config。

### 可提取的方法论

1. **前处理、核心转换、后处理分层**：外部图表先标准化成图片，后端只处理稳定输入，输出再统一修正。
2. **策略模式式双后端**：共同抽象接口隔离不同实现，Auto 模式根据配置与环境选择策略。
3. **配置驱动而非代码分叉**：字体、页面、模板、Pandoc、Mermaid 等由 YAML 控制。
4. **外部依赖边缘化**：Pandoc 和 mmdc 被放在可选边界，核心 CLI 仍有无外部程序的运行路径。
5. **持久化安全独立于转换逻辑**：save 使用临时文件、原子替换和备份轮换，避免生成成功但落盘破坏旧文件。

### 风险提示

README 的后端对比表把 Pandoc TOC 描述为 --toc，而 CLAUDE.md 说明当前为避免冲突使用手工 _apply_toc_field。主架构不受影响，但说明部分用户文档可能落后于实现细节。

### 置信度

**高。** 流程图、目录结构与 Agent 架构约束相互印证。

## 5. C3 设计意图证据卡

**判定：✅ 设计动机和权衡能够从多处材料交叉还原。**

### 核心设计意图

项目不是简单拒绝 Pandoc，而是把自建能力和成熟工具复用组合起来：

- Pure Python 后端解决中文字体、字号、行距、首行缩进、表格样式等精细控制问题，并提供较小文件和低外部依赖路径。
- Pandoc 后端复用成熟 Markdown 解析、代码高亮、章节编号、TOC 和高质量公式处理。
- Auto 模式在 Pandoc 已启用且可用时选 Pandoc，否则回退 Pure Python。
- Mermaid 在进入任一后端前统一渲染，避免每个后端重复实现图表语义。
- 对 YAML 无法覆盖的复杂样式，允许人工调整 DOCX 并保存为 reference_docx，保留面向专业排版的人工出口。

### 为何自建而非只用 Pandoc的推断

材料没有一段直接标题为为什么不用 Pandoc，但证据足以支持以下推断：

1. Pandoc 的中文排版后处理能力不足，常需 reference_docx。
2. Pandoc 是外部程序，存在安装、PATH、分页和参数执行风险。
3. Pure Python 可以直接控制 Word 对象模型，并在 Pandoc 不可用时继续工作。
4. 复杂数学公式和代码高亮又不适合重复造轮子，因此保留 Pandoc 高质量路径。

这是一种 **关键差异自建、成熟通用能力复用、环境不可用时可降级** 的设计策略，而不是单后端技术偏好。

### 权衡是否被诚实记录

是。CLAUDE.md 明确把 Pure Python 数学公式标为实验性，并要求面向用户推荐 Pandoc；README 也记录 Pandoc 长表格空白页、Mermaid 比例和 TOC 冲突等限制。项目没有用统一宣传口径掩盖后端不对称。

### 置信度

**高。** 设计权衡在功能对比、已知限制和不可修改不变量中反复出现。

## 7. C5 AI 协作方法论文档证据卡

**判定：✅ Agent 治理规则是本项目最成熟的方法论组件之一。**

### 治理内容

CLAUDE.md 同时规定了：

- **可派任务**：转换器、渲染器、配置、测试、文档、benchmark 和 CLI 验证。
- **禁止事项**：不触碰被排除的审查/产物目录，不提交 project_status.md，不删除或重命名公开模板，不在未做影响分析时修改 AbstractConverter 公共接口。
- **设计不变量**：配置字段契约、公共返回签名、Mermaid DPI 注入、原子保存与备份轮换。
- **标准命令**：安装、测试、单测、benchmark、代码生成。
- **已知坑位**：公式、Mermaid、Pandoc、Windows 编码、模板默认值。
- **代码智能门禁**：编辑符号前必须做 upstream impact，HIGH/CRITICAL 风险必须告警，提交前必须 detect_changes，重命名必须使用图感知工具。

### 方法论特征

1. **授权边界先于任务执行**：先明确 Agent 可以做什么、不能做什么。
2. **公共契约显式化**：不变量比代码风格建议优先级更高。
3. **变更前后双门禁**：修改前做影响分析，修改后做变更范围检测。
4. **风险分级沟通**：高风险不是静默处理，而是要求向用户告警。
5. **负面知识固化**：把历史坑位写成未来 Agent 的操作约束，减少重复犯错。
6. **事实核验也纳入 Agent 文档生命周期**：CLAUDE.md 自身记录了模型来源、审查得分和事实错误修正。

### 局限

GitNexus 规则很强，但材料未提供一次完整的impact → 编辑 → detect_changes → 测试 → 提交实例，因此当前能够确认的是治理设计，而不是每次变更均严格执行。

### 置信度

**高。** 规则具体、可操作、与项目架构紧密绑定。

## 9. C7 闭环修复证据卡

**判定：⚠️ 能识别出多个发现 → 修复/处置 → 验证闭环，但缺少逐项提交级映射。**

### 已记录的闭环

1. R1-R4 发现约 45 项问题，材料列出 backup 断号、landscape 宽度、Mermaid shell=False 注入等具体例子，并称已修复。
2. R5 发现 16 项数学公式与 MinerU 相关问题，材料称全部修复。
3. CLAUDE.md 审查发现事实错误，文档头记录 61/100 后修正 5 处事实错误。
4. 用户目视验证确认 Pandoc 数学表现较好，同时暴露 Pure Python 渲染、表格分页和默认路径问题。
5. 对不能充分修复的 Pure Python 数学渲染，项目采用标为实验性、公开已知缺陷、推荐 Pandoc、保留原文回退的处置闭环。

### 方法论特征

闭环不只包括改代码直到测试通过，还包括三种终态：

- **修复**：问题被代码或文档变更消除。
- **降级**：后端不可用或能力不足时切换到替代路径。
- **显式接受**：暂时无法普适解决时，记录限制、给出规避方案和人工处理出口。

这种三态闭环比单纯宣称全部修复更可信，尤其适合 DOCX 渲染这类存在平台、字体和 Word 行为差异的问题。

### 缺口

- 没有问题编号、审查条目、提交哈希、修复测试和复验结果之间的一对一链接。
- 16 项全部修复与Pure Python 数学仍有已知缺陷需要解释范围：较合理的理解是审查项已处理，但整体能力仍未达到 Pandoc 品质；材料没有明确列出该边界。
- 用户目视发现的默认路径缺扩展名是否已修复，材料未明确。

### 置信度

**中高。** 可见闭环模式，但审计粒度不足。

## 10. C8 数据溯源证据卡

**判定：⚠️ 具备来源意识和内部定位能力，尚未形成强可复现审计链。**

### 已有溯源机制

- CLAUDE.md 标注生成模型、调用环境、日期、审查模型、评分变化和修正数量。
- 审查追溯表记录轮次、模型、审查角度与关键发现。
- R-分析给出内部 Markdown/JSON 报告路径。
- 项目结构列出 project_status.md、reference_files.md、截图目录、测试 fixture、Schema 和模板目录。
- GitNexus 区块提供仓库上下文、功能簇、执行流程等资源定位方式。
- 转换输出采用原子替换与备份轮换，有利于保留文档产物历史。

### 溯源弱点

1. 审查目录和 pipeline 输出目录被 .gitignore 排除；若没有外部归档，仓库克隆者无法复核这些证据。
2. 没有把审查轮次绑定到 commit SHA、Git tag、CI run、测试报告或 DOCX 校验值。
3. 测试数量出现 17、52、73 三种口径，缺少统计日期 + 统计命令 + 对应版本。
4. GitNexus 指标出现两套快照：审查表为 992 节点、1755 边、34 社区、69 流程；后续区块为 1219 symbols、2173 relationships、92 flows。指标增长可能来自重新索引，但未标注各自提交和索引时间。
5. README、CLAUDE.md 与提示词概述之间存在状态漂移，说明当前没有自动同步的单一事实源。

### 改进建议

建议为每次发布生成一个可提交或可发布的 evidence manifest，至少包含：

- commit SHA、tag、构建时间和工具版本；
- pytest 收集数、通过数、失败数、覆盖率和 CI run；
- 审查模型、提示词版本、原始报告路径及摘要哈希；
- 生成的示例 DOCX、截图和文件 SHA-256；
- GitNexus 索引时间、对应 commit 和指标；

## 11. 跨组件方法论归纳

### 11.1 异构后端的 能力路由

不是强求两个后端功能完全对称，而是先声明各自优势，再通过 Auto 模式、配置和文档推荐进行能力路由。该方法适用于一个实现可控、另一个实现成熟的工具链整合。

### 11.2 配置即契约

DocxPipelineConfig 字段、YAML 模板和 JSON Schema 被视为同一公共契约。模板文件名、字段名和公共方法签名均被列为不可随意修改的不变量。这能把兼容性风险从隐性代码知识提升为显式治理规则。

### 11.3 质量保障覆盖语义正确和视觉正确

DOCX 是渲染型产物，仅有单元测试不足。项目把页面审查和用户目视验证纳入流程，能够发现空白页、图片拉伸、公式显示和分页等难以通过普通断言覆盖的问题。

### 11.4 负面知识是正式资产

数学公式缺陷、DPI、keep_with_next、autofit、TOC 冲突、Windows 编码和发布文件名等历史问题被集中记录。其价值不是解释过去，而是约束未来 Agent 和维护者。

### 11.5 闭环允许多种处置结果

对于确定性问题执行修复；对于环境差异提供后端降级；对于无法普适解决的排版问题公开限制并提供人工出口。该方法避免把尚未完全解决伪装为已完成。

### 11.6 Agent 治理由通用守则下沉到仓库语义

规则没有停留在先测试、别破坏代码，而是指明 AbstractConverter、模板路径、DPI 注入、原子保存等具体高风险对象，并绑定 GitNexus impact 和 detect_changes。说明高质量 Agent 文档应以仓库真实依赖关系为核心。
