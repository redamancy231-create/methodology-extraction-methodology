## 可提取的维度

### 1. 项目元数据

**提取质量**：高——版本历史完整（VERSION 文件 + §14 edit_history），每版有同步脚本。

### 2. 代码/文档组织模式

| 模式 | 证据 |
|------|------|
| **最小核心原则（§1.7）** | 框架自身遵守"加复杂度比减复杂度容易"——新增内容优先外挂为独立文档或衍生项目，而非膨胀主文档 |
| **三件套同步** | .md → .json → .docx 同步链——修改顺序固定，任一件修改后触发异后端交叉验证 |
| **衍生项目提取** | 当框架的子节成熟到可独立使用时，提取为独立仓库（independent-review-toolkit ← §9.2, prompt-tdd-methodology ← §4.1.1）——防止主文档膨胀 |
| **LSP 优先策略** | CLAUDE.md 定义了 LSP vs grep 的五级选择策略——这是 20 个源项目中最精细的 Agent 工具使用规范 |

**提取质量**：高——最小核心原则和三件套同步是有意设计的架构约束。

## 不可提取/弱提取的维度

### 7. 闭环修复模式

**提取结果**：**部分可见。** 版本演进本身就是一种闭环——v1.5.1 的冻结期教训写入了 v1.6.x 的 §1.7。但缺乏显式的"发现 X → 修复 X → 验证修复"闭环记录——审查发现通常被汇总后批量修复，单个修复的验证过程不可追踪。

这与其他自有项目一致——闭环修复是系统性短板，但框架的版本管理和 OPEN 项制度至少保留了"什么还没修"的记录。

### 8. 数据溯源

**提取结果**：**框架定义了但未在自己的主文档上执行。** 数据溯源四级分类 [R][S][E][P] 在 ma-case-study-pipeline 中应用，但框架自身的证据标注（[S]/[E]/[I]/[J]/[Sp]）是方法论来源分类，不是数据溯源。不适用。

## 独特方法论贡献（相比其他自有项目）

### 模式 1: 最小核心原则 + 衍生提取

框架的 §1.7 规定"新增内容优先外挂，而非膨胀主文档"。当框架的子节（§9.2 独立审查、§4.1.1 对照实验）成熟到可独立使用时，被提取为独立仓库（independent-review-toolkit、prompt-tdd-methodology）。**这是"方法论→工具"的进化路径**——从文档描述到可执行 CLI。

这是 所有源项目中唯一的"方法论文档主动约束自身复杂度"的案例。

### 模式 2: 审查独立性硬约束

"禁止用同一后端模型审查框架——同一后端换 CLI 壳 = 伪独立"——比其他项目的铁律 2（"无人审己"）更严格。铁律 2 禁止同一模型审查自己的产出，框架的硬约束连"同一后端的不同实例"都禁止——因为后端共享训练数据/RLHF/系统提示，换壳不等于独立。

### 模式 3: 成熟度自我评估 + OPEN 项制度

67 项元审查清单对框架的每个组件标注成熟度——哪些已验证、哪些仅理论推导、哪些从未试跑。OPEN 项显式登记"已知但未解决的问题"。这是 所有源项目中**唯一一个对自己的"不知道什么"做了结构化记录**的项目。

### 模式 4: 冻结期教训的制度化

v1.5.1 冻结期的核心教训（"暂停加新功能，先把已有组件的测试/验证补齐" → 但实际执行中被绕过了）被写入了 §1.7 作为永久操作约束。**方法论教训从"一次性的复盘条目"变成了"规则化的操作约束"**——这正是方法论提取框架追求的 C7（闭环修复）的最高级形式。
