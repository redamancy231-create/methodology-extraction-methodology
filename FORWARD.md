# 推进路线图

## 当前状态

**v0.1 trial · CLOSED · 6/10 可信度**

经过两轮 GPT-5.6-Sol 异后端审查（2026-07-24），项目叙事已基本自洽，但方法论本身仍处于试行阶段。以下是可确认的事实和已知局限：

### 已经可以相信的

- 22 个源项目的分类清单（`source_projects_summary.md`）经过逐项核实
- 11 张公开证据卡（`explorations/evidence_card_*.md`）全部可定位
- 三份结构化清单（`evidence_card_manifest.json`、`review_manifest.json`、`experiment_manifest.json`）提供了机器可核验的数字
- 10 轮审查链在叙事层面可追溯（审查报告目录见 `review_manifest.json`）
- 框架的核心方法（多模型独立审查、提取者效应检测）在 9 项目对照实验中得到了实证支持

### 还不能相信的

- 审查发现数（58 → 87 → 88？）缺少逐项登记表和统一的纳入/排除规则
- P1-P8 模式目录与原始 Phase 4 目录（`synthesis/phase4_validated_pattern_catalog.md`）存在 ID 冲突
- 框架的通用性未经外部作者/外部操作者验证——全部提取由同一人执行
- G5 审计结论中的"对失败诚实"达标模式未列入正式模式目录

## 已知局限（分层）

### 第一层：数字漂移（可自主修复）

**根因**：README、DEV_LOG、CLAUDE.md 中的数字手工维护，必然漂移。

**修复路径**：
1. 从三份 manifest JSON 自动生成 README 中的统计数字
2. 建立 `finding_register.json`，逐项登记每条审查发现（来源、严重性、处置、修复 commit）
3. 在 CLAUDE.md 增加 corpus 版本表（v1=12、v2=19、v3=22），标注各阶段包含哪些项目

**工作量**：1-2 个专注会话。

### 第二层：框架内部矛盾（需要动交付物）

**根因**：框架正文（`framework_output/main/`）在闭合后曾被多次修订，但模式目录（`synthesis/`）未同步。P4/P7 的定义在两处不同。

**修复路径**：
1. 以 `synthesis/phase4_validated_pattern_catalog.md` 为权威，统一模式 ID
2. 在 `source_projects_summary.md` 中标注"此表为引用摘要，权威定义见 Phase 4 目录"
3. 如确实需要新 ID，分配 P9+ 而非复用旧 ID

**工作量**：1 个会话。**注意**：框架正文是 v0.1 trial 的历史快照——修改它会改变交付物。如果希望保留历史完整性，可以在 `source_projects_summary.md` 中加映射表而非修改原文。

### 第三层：提取者污染（需要外部贡献）

**根因**：22 个源项目的筛选、证据卡填写、模式提取、G5 审计全部由同一人执行。GPT-5.6-Sol 的 9 项目对照实验已经量化了提取者效应（外部项目全部抬升、自建项目无一抬升），但效应的大小和方向只在一个模型对上测量过。

**验证路径**（需要外部贡献者）：
1. 由不同操作者独立提取 2-3 个源项目
2. 由不同操作者独立执行 G5 审计
3. 比较操作者间一致性（不仅是模型间一致性）
4. 贡献外部反例——在哪些项目上提取框架完全失败

**不需要外部贡献的部分**：
- 补充第 3-5 个模型（如 Qwen、Claude）的对照提取
- 在现有 11 张证据卡上标注 DeepSeek 提取 vs GPT-5.6-Sol 提取的差异字段
- 将提取差异量化为每个组件的分歧率

### 第四层：框架泛化性（主要瓶颈）

**根因**：源项目中 11 个是分析别人的项目（外部分析），但"分析别人的项目"本身也是一种特定类型的 AI 协作方式——它不等于你自己的项目使用了 AI 协作。

**修复路径**：
1. 寻找 ≥1 个真正由他人独立完成的 AI 协作项目，由其作者自己记录过程（而非由你分析后记录）
2. 对该项目执行提取，对比提取结果与作者自述的差异
3. 如果找不到——框架的定位声明应降级为"个人方法论提取工具"，而非"跨用户通用框架"

## 不应做的事情

- **不要继续增加源项目数量而不提升提取独立性**。n=22 已经足够揭示模式——继续扩大到 n=30 不会改变"同一操作者"的局限。
- **不要在没有分支保护的情况下修改框架正文**。`framework_output/main/` 是 v0.1 trial 的历史记录。如需修订，建分支、留 diff、记录版本。
- **不要删除或修改 `reviews/` 中的原始审查报告**。它们是审查链的原始证据。
- **不要追求"完整提取"或"A 级证据"作为目标**。框架自身的 G5 审计结论就是"无一模式达标"——追求完美的自我评分恰恰违反了"对失败诚实"这个唯一达标的模式。

## 如何贡献

如果你不是原作者，想 fork 并推进：

1. 先读 `FORKING.md`——它覆盖了四层修改方向（叙事层/语料层/方法论层/框架层）
2. 最有价值的贡献方式：
   - 贡献你自己的 AI 协作项目作为源项目（含完整 CLAUDE.md + DEV_LOG + 审查链）
   - 作为独立操作者重提取现有源项目
   - 贡献反例——你的项目为什么提取框架不适用
3. 提 PR 前先读 `CONTRIBUTING.md`
