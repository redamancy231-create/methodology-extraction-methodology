## 1. 有效票数分析（Kohli 2026-05 "Nine Judges, Two Effective Votes"）

### 论文核心发现

Kohli（2026-05）测试了 9 个前沿 LLM 组成的评审团（来自 7 个模型家族），发现：
- 有效独立票数仅 **~2 票**（独立比率 ~24%）
- 评审团实际准确率比独立投票假设低 **8-22 个百分点**
- **单一最佳评委追平甚至超过整个评审团**
- 增加评委或使用更智能的聚合算法（Dawid-Skene EM, accuracy-weighted voting）无法弥补超过 11% 的差距——瓶颈是**评委之间的关联性**，而非算法

### 应用到本项目

本项目使用 **4 个审查后端**（GPT-5.5 / Kimi-K2.7 / Qwen3.7-Max / GLM-5.1）加 DeepSeek-V4-Pro 作为执行者。按 Kohli 的模型推算：

| 指标 | Kohli 测量值 | 本项目估算 |
|------|-------------|-----------|
| 审查后端数量 | 9 | 4 |
| 模型家族数量 | 7 | **5**（OpenAI/Moonshot/Alibaba/Zhipu/DeepSeek——全部来自不同提供商） |
| 有效独立票数 | ~2 | **~2-3**（跨提供商多样性可能略高于 Kohli 的测试集） |
| 独立比率 | ~24% | 估计 30-40%（更分散的模型来源） |

**关键推论**：

1. **你的 4 审查后端架构不是过度设计——它恰好接近有效上限。** 如果 Kohli 的结论成立（9→2），那么 5 模型角色槽位中 4 个用于审查已经是最优配置。加更多模型不会带来增量收益，减少模型则会直接降低审查覆盖率。

2. **Phase 5 零重叠是预期行为，不是统计巧合。** 两个有效独立票对同一份文档产生的发现集合，在理论上有相当概率不重叠——因为它们的"错误敏感性空间"近似正交。Codex 17 项 + Qwen 16 项 = 33 项零重叠，恰好符合"两个有效独立审查者"的预期产出模式。

3. **但这也意味着：你的 10 轮审查 ≠ 10 个独立视角。** 实际有效视角可能只有 2-3 个。这解释了为什么审查评分在不同后端之间高度相关（Codex 5.5-7.7, Kimi 5.9, Qwen 5.5-6.3）——它们的评分波动主要来自任务难度差异，而非审查者独立性。

### 局限性

本分析是**定性外推**——Kohli 测量的是分类任务的投票独立性，而本项目是多维审查任务。审查任务的"独立性"可能有不同的结构（错误敏感性空间的维度更高），因此 2 票可能是一个**保守下界**。但精确估计需要对本项目的审查输出做成对误差相关性分析，这超出了本附录的范围。

## 2. 审查发现分类学重分析（Nájera et al., 2026-05 "When Models Disagree"）

### 论文核心发现

Nájera 等（2026-05）提出将多模型分歧分为四类：

| 类型 | 定义 | 信号含义 |
|------|------|---------|
| **共识** (Consensus) | 所有模型给出相同判断 | 高置信度——任务有明确的正解 |
| **二元分裂** (Binary split) | 评审分成两个阵营 | 中置信度——存在两种合理的解释框架 |
| **深度分歧** (Deep disagreement) | 每个模型给出不同的判断 | 低置信度——任务的解释空间高度开放 |
| **立场分歧** (Stance disagreement) | 分歧源于模型的设计立场（安全偏好/训练分布） | 系统性偏差——分歧反映的是模型属性，不是任务属性 |

论文核心主张：**分歧类型本身携带信息**——不同类型的分歧指示不同程度的解释复杂性和不同的质量风险。

### 应用到本项目：58 项发现的四类重分类

对本项目累计 58 项审查发现（Phase 1-8 全部审查产出）进行重分类：

| 类型 | 数量（估计） | 代表性发现 |
|------|:-----------:|-----------|
| **共识** | ~8（14%） | 所有审查后端一致指出的重大事实错误（如 Phase 5 Codex #1 "12 个闭合包含搁置项目"） |
| **二元分裂** | ~22（38%） | 两个后端独立发现、其余后端未覆盖的归类/自相矛盾问题（如"操作化"措辞越界：Codex 完全错过，Qwen 发现） |
| **深度分歧** | ~20（34%） | Phase 5 双轮审查零重叠区域——Codex 17 项（事实/归类/完备性）+ Qwen 16 项（G5 忠实度/逻辑一致性/操作可行性/反例测试），在当前编码规则下未观察到重叠 |
| **立场分歧** | ~8（14%） | 评分维度和严格度差异（Kimi 最严 5.9, Codex 最宽 7.7）——反映的是模型的评分偏好和训练分布，不是任务属性 |

**关键发现**：

- **34% 的深度分歧率意味着：需要 ≥3 个审查后端才能实现合理的审查覆盖率。** 用 2 个后端，会漏掉大约三分之一的潜在发现问题。这验证了项目使用 4 审查后端的设计决策。
- **立场分歧占 14%**，且主要表现为评分差异而非发现差异（不同后端发现的**内容**极少重叠，但**评分**高度相关）。这与 Kohli 的结论一致——"多模型"的价值在于**发现空间的互补**，而非**判断权重的聚合**。

### 一个假设（待验证）

Nájera 的分类法是否适用于代码/文档审查场景（而非其原文的公共评论分析场景）？本项目的重分类表明**适用，但需要调整粒度**——代码审查中的"共识"通常对应 P0 级硬错误（如 python IndentationError），而"深度分歧"对应的是措辞精准度、结构一致性和方法论归属等软性问题。

## 3. 行为纠缠的非正式评估（Kuai et al., 2026-04 "Behavioral Entanglement"）

### 论文核心发现

Kuai 等（2026-04）提出了两个信息论指标量化 LLM 之间的"行为纠缠"：
- **难度加权行为纠缠指数**（Difficulty-Weighted Behavioral Entanglement Index）：放大简单任务上的同步失败
- **累积信息增益**（Cumulative Information Gain）：捕捉错误响应的方向性对齐

测试 18 个 LLM（来自 6 个模型家族），发现广泛的行为纠缠：
- 纠缠度与 LLM-as-a-Judge 管道中的过度认可偏差正相关（Spearman ρ = 0.64–0.71, p < 0.01）
- 去纠缠验证器集成可将准确率提升 **4.5%**

### 应用到本项目：你的模型选择策略隐式降低了纠缠

本项目的模型选择——**五个不同提供商的模型**（Moonshot / Zhipu / OpenAI / Alibaba / DeepSeek）——恰好是一种"去纠缠"策略：

| 设计选择 | 纠缠降低效果 |
|---------|------------|
| 全不同提供商 | ★★★ 最强——不同训练数据、不同 RLHF、不同系统提示 |
| 角色轮换（审查"自己没做过的环节"） | ★★ 中——降低同一模型在相邻 Phase 的自我一致偏差 |
| 零卷入评分官（Qwen，未参与任何前序环节） | ★★ 中——消除出题人偏差 |
| 开卷/盲答对照 | ★ 弱——控制答案键泄漏，但两个评分阶段用的是同一模型 |

**一个可测试的预测**：如果对本项目的 58 项发现做成对纠缠分析（只取同一份审查对象、不同后端独立审查的发现集），预估有效独立票数在 **2.0-2.5 之间**，高于 Kohli 的 ~2.0（因为跨提供商多样性更高），但远低于名义票数 4。

## 4. 协作框架的系统对比（Oda, 2026 "Human-Led AI Co-Creation"）

### 论文核心框架

Oda（2026）提出四角色框架：

```
主 AI 起草 → 多模型并行审查 → 迭代跨模型对比 → 人类最终裁决
```

识别了四种失败模式：长线程退化、精炼压力、静默摘要化、声音失真。

### 与本项目流水线的系统对比

| 维度 | Oda (2026) | 本项目（2026-05） |
|------|-----------|-----------------|
| **阶段数** | 隐式（4 角色循环） | 显式（8+2 Phase，带分叉汇合） |
| **角色定义** | 功能性（起草/审查/对比/裁决） | **槽位制**（角色是槽位，模型填槽，每项目重配） |
| **独立性保障** | 人类最终裁决（单点保障） | **三条铁律**（无人审己/出题评分分离/零卷入评分官） |
| **可审计性** | 低——无 prompt/config 双文件 | **高**——每阶段 prompt.md + config.json |
| **迭代机制** | 多轮跨模型对比（可能无限循环） | Phase 9 闭环（诊断→修复，单次循环） |
| **实验验证** | 案例演示（n 未报告） | 对照实验（开卷/盲答, n=1 案例研究） |
| **失败模式记录** | 4 种系统分类 | 3 处论文硬伤 + Phase 4 自评 106% + Phase 8 JSON 语法错误 |

**关键差异**：

1. **Oda 的框架依赖人类作为唯一的独立性保障**——如果人类也犯错（如本项目 Phase 4 自评 106%），整个系统没有内部纠错机制。本项目的三条铁律在**人类介入之前**就建立了结构性的独立视角。

2. **Oda 的迭代是开放的**——"多轮对比→修正→再对比"可能无限循环。本项目的 Phase 9 是**单次闭环**——诊断要么修复、要么封存，不承诺迭代到完美。这是对"精炼压力"失败模式的显式防御。

3. **Oda 不要求可审计性**——没有 prompt 保存、没有 config 版本控制、没有审查链追踪。本项目将可审计性作为**基础架构**（铁律 1），这使得后续的质量分析和相位迭代成为可能，而非依赖记忆和善意。

### 互补性

两个框架在学术价值上**互补而非竞争**：
- Oda 的框架更**通用**——适用于任何 AI 辅助写作，不需要技术基础设施
- 本项目的框架更**严格**——适用于需要可复现性和独立验证的场景（学术论文、正式报告）
- Oda 的失败模式分类**可以直接注入**本项目的 Phase 0 风险登记册
- 本项目的 audit trail 设计**可以升级** Oda 框架的可审计性

## 5. Swiss Cheese Model（CrossCheck 对照）

> 来源: [sburl/CrossCheck](https://github.com/sburl/CrossCheck)——2026-07-23 外部压力测试 #3

### CrossCheck 的核心设计

CrossCheck 明确引用了航空安全领域的 **Swiss Cheese Model**（Reason, 1990）作为其多层质量保障设计的理论基础：每一层安全屏障都有漏洞（"奶酪上的孔"），但多层不相关的屏障叠加后，故障穿透所有层的概率趋近于零。

CrossCheck 的六层：
```
Plan → Build → Verify → Review → Merge → Triage
       (hooks)  (pre-PR)  (peer)   (branch)  (backlog)
```

### 本项目的隐性 Swiss Cheese Model

本项目的 QA 架构在 CrossCheck 的视角下，恰好也是一种 Swiss Cheese Model——只是从未被这样命名：

| 层 | 本项目 | CrossCheck 对应 |
|------|------|------|
| **第 1 层** | Phase 2 领域专家撰前审核 | （无——CrossCheck 不区分撰前/撰后审查） |
| **第 2 层** | Phase 5A/5B 交叉双盲审 | Phase 4 Review（peer model review） |
| **第 3 层** | Phase 7 设计回溯 | Phase 6 Triage（发现→规划修复） |
| **第 4 层** | Phase 8 答辩模拟 | （无——CrossCheck 无答辩/评分环节） |
| **第 5 层** | Phase 9 闭环修复 | Phase 5 Merge（验收合入） |
| **第 6 层** | 开卷/盲答对照实验 | （无——CrossCheck 无对照验证机制） |

**关键差异**：

| 维度 | CrossCheck | 本项目 |
|------|-----------|--------|
| **合规执行** | Git hooks——被动，不需要 agent 记住规则 | 铁律 2/3——主动，依赖 agent 自觉 |
| **漏洞类型** | 漏掉 pre-push check（人为跳过 hook） | 同一后端换壳伪装独立（agent 绕过铁律） |
| **理论命名** | ✅ "Swiss Cheese Model"——有学术根基 | ❌ 无命名——设计存在但未被理论化 |
| **可迁移性** | `git clone` → 安装 hooks → 即用 | 复制 playbook → 填模板 → 人工分配角色 |

### 对本项目的启示

1. **给设计命名**：本项目的多层 QA 本质上就是 Swiss Cheese Model 在 AI 协作场景中的应用。在框架中显式引用这一理论框架，可以为"为什么需要 ≥3 层独立审查"提供学术上有根基的解释——而不仅仅是"我们试了，发现有效"

2. **被动合规 vs 主动自觉的互补**：CrossCheck 的 hooks 模式强制执行编码规范（密码检测、提交格式），但不能强制执行"审查独立性"（agent 仍可能用同一模型自审）。本项目的铁律模式强制执行独立性，但不能防止操作层面的遗漏（如 Phase 8 JSON 语法错误从 v1.0 存在到被发现）。**两种策略的盲区恰好互补**

3. **每层应该有不同的"漏洞类型"**：Swiss Cheese Model 的关键假设是不同层的漏洞不相关（"uncorrelated holes"）。本项目的 QA 层虽然多层，但是否真的不相关？——盲审和答辩都依赖"模型能发现问题"（同一类漏洞：LLM 审查盲区），而 CrossCheck 混合了 hooks（机械漏洞）和 peer review（认知漏洞）。**增加一层机械验证（类似 CrossCheck 的 hooks）可能比再增加一层认知审查（再多一个审查模型）更有效**

## 6. 综合评估

### 四条外部验证

| 论文 | 验证了什么 | 修正了什么 |
|------|-----------|-----------|
| Kohli (2026-05) | 4-5 模型架构是最优配置，不是过度设计 | 10 轮审查 ≈ 2-3 个有效独立视角，不应声称"10 轮独立" |
| Nájera (2026-05) | Phase 5 零重叠是"深度分歧"类型的预期表现 | 58 项发现可系统分为 4 类，34% 为深度分歧——说明 ≥3 后端的必要性 |
| Kuai (2026-04) | 全不同提供商的模型选择是隐式去纠缠策略 | 纠缠度虽低但非零——不能声称"完全独立" |
| Oda (2026) | 本项目的铁律体系比学术界最接近的框架**更严格** | Oda 的失败模式分类可注入本项目的 Phase 0 |

### 项目定位修正

基于外部验证，本项目在学术坐标系中的位置更加明确：

> **在 2026-07-23 对 GitHub（9 个搜索方向）和学术文献的系统搜索中，未检索到从多个 AI 协作项目中系统提取可复用方法论文献的同类工作。搜索存在已知局限（GitHub 无法索引微型仓库，学术搜索受关键词选择影响），不排除相关工作存在但未被发现。本项目的核心方法论主张（多模型独立审查的价值、模型间的错误空间互补性、结构化的独立性保障）已被 2026 年独立学术文献验证。但其提取产物受限于 n=12 同作者样本，尚未达到跨用户泛化的稳定门槛。**

### 如果本项目继续（不承诺）

以下方向基于外部验证确定：

1. **做 Kohli 式的成对误差相关性分析**——对本项目已有的审查数据集做定量独立性测量，产出第一个"代码/文档审查场景下的 LLM 行为纠缠"数据点
2. **用 Nájera 分类法重新标注全部 58 项发现**——产出规范化的跨项目审查发现数据集，可被后续研究复用
3. **与 Oda 框架做受控对比实验**——同任务、同评判标准，测试"铁律隔离" vs "人类单点保障"的审查覆盖率差异
4. **扩样本**——在至少 3 个不同作者的 AI 协作项目上重新跑提取协议
