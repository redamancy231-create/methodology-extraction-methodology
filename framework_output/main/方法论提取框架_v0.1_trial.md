## §0 使用须知

### §0.0 方法论定位

本框架的**方法论定位**为「**方法论提取实践的系统化总结**」，非方法论创新。具体而言：

- **原创性贡献**（增量）主要集中于：双轴审查 SOP 的跨项目推论轴（§5.2）、稳定性标签体系（§0.2，区分证据强度与证据来源类型）
- **系统化贡献**（整合）包括：项目筛选 rubric（§1）、四阶段提取流水线（§2）、三级分流决策模板（§3）、对抗验证流程（§4.1）
- **非原创**（现有实践的系统化）：准入条件检查、证据收集流程、片段提取模板等来自 systematic review、lessons learned 流程、meta-analysis 等成熟方法的适配

与现有方法论提取文献的关系：
- 与 **systematic review** 共享：结构化筛选、质量评估、证据综合
- 与 **lessons learned 流程** 共享：失败/教训分类、根因分析
- 与 **meta-analysis** 共享：跨项目效应量估计、异质性分析
- **独有增量**：对抗验证（专门否定者推翻候选模式）、复编码门禁（异后端独立重编码高风险维度）、稳定性标签（区分 \([S]/[E]/[I]/[Sp]\) 证据来源类型）

全组件试行版的诚实标注反映了当前证据状态，但同时也意味着用户不能期望稳定复现——本框架更像「方法论探索报告」而非「成熟方法论工具」。待 ≥3 个独立项目验证后方可升级部分组件为稳定。

### §0.1 G5 审计核心结论

本框架 v0.1 的 5 个组件经 `synthesis/phase4_traceability_audit.md` 严格审计，审计口径为"12 源项目直接证据"（不含本元项目过程证据）。结论：

> **当前无一模式（P1-P6, P8）满足 ≥3 源项目直接证据的稳定组件门槛。**
> 
> 满足项目数但仍不能稳定化：P2（两类错误分化被证伪）、P3（事后确认偏误）。
> 不满足项目数：P1（2 源项目）、P5（2 源项目复编码）、P6（核心比较证据来自本元项目）、P8（0 效率测量）。
> 已拒绝：P4（6 条独立否决理由）、P7（关系方向反了）。

因此本版所有组件均标注为**试行协议**，待未来 ≥3 个独立项目验证后方可升级为稳定组件。

### §0.2 标签体系

本框架使用四类稳定性标签，定义如下：

| 标签 | 符号 | 门槛 | 含义 |
|------|------|------|------|
| **稳定组件** | `[S]` | ≥3 源项目直接证据 + 无证伪反例 | 可无条件应用于新项目，预期稳定复现 |
| **试行协议** | `[E]` | ≥1 源项目证据但不足 3，或虽 ≥3 但命题被削弱/反例未解决 | 可尝试应用，须记录效果并独立评估；不预设升级预期 |
| **审查门禁** | `[I]` | 来自本元项目过程证据或审查发现 | 本元项目总结的流程纪律；不宣称跨项目稳定性。标注为 [I] 的规则应在未来独立项目中验证后再考虑升级 |
| **工具注记** | `[Sp]` | 工具能力观察，非方法论主张 | 适用条件明确时可使用，注意配套约束 |

**两套分类系统关系**：本框架使用两套独立分类——(a) **模式强度**（§2.1 Phase C，7 级：A/A-/B/B-/C+/Sp/REJECTED）衡量证据量；(b) **稳定性标签**（§0.2，4 类：[S]/[E]/[I]/[Sp]）衡量证据来源类型和可推广性。二者可独立变化：强度高的模式可能因证据来自本元项目而标 [I]（如 P6 A-→[I]），强度低的模式可能因有独立源项目证据而标 [E]（如 P1 C+→[E]）。冲突时以稳定性标签为准（标签决定使用权限）。

当前版本（v0.1-trial）：**无一组件标 `[S]`**。全 5 组件最低标 `[E]`，部分子条款标 `[I]` 或 `[Sp]`。

本框架是 [AI协作项目全生命周期框架 v1.5.1](https://github.com/redamancy231-create/ai-collaboration-framework) 的 **Protocol 3 首次真实试跑产物**。关系如下：

- **父框架提供**：项目规模分档（S/M/L）、独立审查要求、证据等级体系 [S]/[E]/[I]/[J]/[Sp]、Dev Log 维护纪律、闭合规范
- **本框架将回写**（Phase 8 待执行）：Protocol 3 预登记指标 C1-C8 实测数据、Retrospect 报告（≥1 失效 + ≥1 意外）、框架成熟度评估表更新
- **递归自检**：父框架自身的开发史（v1.0→v1.5.1）作为第 13 个提取源候选（Phase 6，本版未执行，不计入 G5 审计的"源项目直接证据"）

### §0.4 适用范围与源项目选择偏差

- **适用项目类型**：AI 协作完成的分析/研究/评估类项目（含量化研究、文献综述、代码审计、工具评估）
- **不适用**：纯工程实现项目（无分析/判断/方法论成分）、单次问答（非项目形态）、创作类项目（设计/写作/艺术）、决策支持类项目（战略规划/商业决策）
- **提取时机**：项目闭合后（含闭合记录 + 终期总结），活跃项目仅做边界标注不进入完整提取

**源项目选择偏差**（已知局限）：

本框架基于 12 个 AI 协作项目提取方法论，样本存在以下选择偏差：

| 偏差类型 | 具体情况 | 对方法论可迁移性的影响 |
|---------|---------|---------------------|
| **项目类型同质** | 全部为 AI 协作分析/研究/评估类项目，无工程实现/创作/决策支持类 | 框架可能不适用于其他类型项目 |
| **操作者单一** | 所有项目由同一操作者（用户 + DeepSeek-V4-Pro）在 72h 窗口内完成大部分工作 | 操作者依赖性可能影响方法论可迁移性；不同操作者的项目可能呈现不同模式 |
| **规模偏倚** | 以中大型项目为主（产出 ≥3 文件，有闭合记录和审查报告） | 小型/轻量项目的方法论提取需求可能不同 |
| **闭合状态偏倚** | 大部分项目已闭合/归档，方法论提取在事后进行 | 事后提取可能遗漏过程中的实时方法论决策 |
| **AI 模型偏倚** | 源项目主要由 DeepSeek-V4-Pro 执行，方法论片段可能带有该模型的系统性特征 | 其他模型执行的项目可能呈现不同的方法论片段分布 |

**适用性自检**（使用框架前必读）：

用户需评估目标项目与源项目的相似性：
1. 项目类型是否匹配？（分析/研究/评估类 → 适用；工程实现/创作/决策支持 → 谨慎）
2. 项目规模是否匹配？（中大型 → 适用；小型 → 走精简路径）
3. 操作模式是否相似？（AI 协作 + 多轮迭代 → 适用；一次性生成 → 谨慎）
4. 是否有独立审查报告？（有 → 完整提取；无 → 降级为片段提取）

若 ≥2 项"谨慎"，建议先走 2 小时试点路径（§0.5）验证适用性，再决定是否投入完整流程。

### §0.5 最小可行路径

**使用本框架的前提条件**：

| 条件 | 要求 | 不满足时的替代方案 |
|------|------|------------------|
| 1. 项目闭合记录 | 用户项目必须有闭合记录或终期总结（否则无法判断终态） | 活跃项目仅做边界标注，不进入完整提取 |
| 2. 独立审查报告 | 用户项目**建议**有 ≥1 份独立审查报告（否则证据包缺失验证层，§2.1 A3） | 降级为「片段提取」路径（仅 Phase A-B，不做跨项目综合）；或标注"验证层缺失"后继续，但提取结论降置信度 |
| 3. 多后端可用 | 用户**建议**有 ≥2 个 AI 后端可用（复编码门禁 §5.4 要求异后端；对抗验证 §4.1 要求 ≥2 个否定者） | 单后端：走「单后端加强版」（多轮审查 + 严格事实核查替代异后端审查）；对抗验证改为作者自查加强版（自评 + 证据等级限定在 B 或以下） |
| 4. 源项目文件完整 | 目标项目的 CLAUDE.md ≥20 行、闭合记录可读、关键产出文件可定位 | 先补齐 CLAUDE.md 再评估；≤2 个产出文件降级为轻量项目 |

框架全文 5 组件 + 4 附录，不同时间预算下的推荐路径：

| 时间预算 | 路径 | 产出 |
|:------:|------|------|
| **2 小时** | §1.1（准入条件）→ §2.1 Phase A-B（证据收集+片段提取）→ §2.2 证据卡模板（填写）→ 停止 | 单项目证据卡（含 ≥3 个方法论片段） |
| **半天** | 上述 + §3.1 三级分流（对片段分类）→ §2.1 Phase C 步骤 1-2（聚类+对抗验证）| 单项目完整提取 + 分流标注 |

| **完整** | 全文（多个项目并行提取 + §4 跨项目综合 + §5 全 SOP）| 多项目综合框架 |

> **首次使用者**建议走 2 小时路径完成一个熟悉项目的试点提取，再决定是否投入完整流程。

### §1.1 最低准入条件（全部满足方可进入提取）

| # | 条件 | 检查方式 | 不满足时 |
|---|------|---------|---------|
| E1 | 项目 CLAUDE.md 存在且可读 | `Read` 验证；内容 ≥50 行（完整）/ 20-49 行（需补齐）/ <20 行（排除） | <20 行标记为"入口不足"排除；20-49 行标记为"需补齐"后评估 |
| E2 | 项目有闭合记录或终期总结 | 搜索 `闭合`/`CLOSED`/`终期` 关键词 | 活跃项目仅做边界标注；搁置项目标记状态后评估 |

**边界案例处理**：

- **活跃项目**（如 BDC2026）：标注为活跃边界，降低归纳权重（折扣系数 0.5-0.7），仅提取已闭合子任务的方法论片段
- **搁置项目**（如多模型对照实验）：标注搁置状态，提取冻结协议中的方法论资产，不提取未验证假设
- **轻量项目**（如 claude-code-ultimate-guide）：若无独立审查报告，标记为"轻量"，仅提取可直接引用的操作规则

### §1.2 提取价值评估（准入后评分）

对满足 §1.1 的项目，按以下维度评估提取价值（1-3 分/维度）：

| 维度 | 1 分（低） | 2 分（中） | 3 分（高） |
|------|----------|----------|----------|
| **方法论密度** | 1-2 个可迁移片段 | 3-5 个可迁移片段 | ≥6 个可迁移片段 |
| **失败教训丰富度** | 无记录的系统性失败 | 1-2 个命名失败模式 | ≥3 个命名失败模式 + 根因分析 |
| **独立审查深度** | 0-1 轮审查 | 2 轮审查 | ≥3 轮异后端审查 |
| **格式成熟度** | 嵌入式（经验嵌在项目文档中） | 独立提取文档存在 | 独立提取 + 双件 + 审查修正记录 |
| **源项目质量** | 综合评分 <5.0 或争议大 | 5.0-7.0 | ≥7.0 |

**方法论密度预判指南**（用于在提取前快速估计"可迁移片段"数量）：

扫描项目文档中的以下特征关键词，每类命中 ≥1 个即可能含 ≥1 个可迁移片段：
- **失败/教训类**：`教训` `陷阱` `已知问题` `失败` `事故` `复盘` `不要`
- **跨项目关联类**：`与.*对标` `镜像` `跨项目` `pattern` `模式` `迁移` `复用`
- **方法论自觉类**：`方法论` `阶段` `流水线` `checklist` `门禁` `审查` `协议`
- **反例/边界类**：`不适用` `边界` `例外` `限制` `仅当` `前提`

**"可迁移片段"客观计数规则**（Phase 7.5 GLM-5 审查 Finding 2 修正）：

一个文本段落计为 1 个"可迁移片段"，须同时满足以下 3 个条件：
1. **跨项目关键词**：包含 ≥1 个上述四类关键词
2. **来源明确**：可定位到具体文件路径 + 行号范围，或项目文档中的明确章节
3. **边界条件描述**：含 ≥1 个条件性表述（"仅当"/"不适用"/"前提"/"限制"/"边界"/"例外"/"注意"），即该片段描述了什么情况下适用、什么情况下不适用

缺少任一条 → 不计入可迁移片段。按此规则重新计数时可能导致 rubric 分数比预判指南的粗略扫描低 1-2 分——预判指南用于快速筛选，计数规则用于正式提取时的精确评分。

**提取优先级判定**：

## §2 提取流程标准化 `[试行协议]`

> **稳定性**: `[E]` — 流程骨架来自 7 份已有提取产物的共同元素（Phase 3 对比矩阵），但完整流水线仅在本元项目执行 1 次，未经独立项目复现。

### §2.1 四阶段提取流水线

```
Phase 0: 项目筛选    ──→  Phase A: 证据收集  ──→  Phase B: 片段提取  ──→  Phase C: 跨项目综合  ──→  Phase D: 交付与验证
   (0.5 loop)              (1-2 loops)            (2-3 loops)              (2-3 loops)                (1-2 loops)
```

#### Phase 0: 项目筛选（分流过滤）

**目标**：在投入完整证据收集前，先判定项目是否值得提取。

| 步骤 | 操作 | 产出 |
|------|------|------|
| 0.1 | 按 §1.1 最低准入条件检查 | 准入/排除/边界标注 |
| 0.2 | 按 §1.2 提取价值评估 rubric 评分 | 高/中/低优先级 |
| 0.3 | 按 §3.1 三级分流预判（项目级，非片段级） | 预分流标注 |

> **接口**：Phase 0 的输出作为 Phase A 的输入过滤器——仅高/中优先级项目进入完整 Phase A 五层证据包；低优先级项目走精简通道（仅 A1+A2）。

#### Phase A: 证据收集

**目标**：建立项目的完整证据地图。

| 步骤 | 操作 | 产出 |
|------|------|------|
| A1 | 读取 CLAUDE.md + 项目 Spec（如有） | 项目宪法摘要 |
| A2 | 读取闭合记录/终期总结 | 终态结论 + 已知局限 |
| A3 | 读取独立审查报告（全部轮次） | 审查发现清单 + 修正记录 |
| A4 | 搜寻方法论相关章节（经验总结/教训/已知陷阱/跨项目关联） | 候选片段原始文本 |
| A5 | 填写证据卡 §1（证据来源清单） | 证据来源表 |

**最小证据包协议**（来自 HG-0 强制修订 M1）：

每项目至少收集以下 5 层证据：
1. 项目 CLAUDE.md（入口层）
2. 闭合记录或终期总结（终态层）
3. ≥1 份独立审查报告（验证层）
4. 方法论相关章节原文（内容层）
5. 关键文件路径索引（可复现层）

若任一层缺失，在证据卡中明确标注缺失原因和替代方案。

#### Phase B: 片段提取

**目标**：从项目中提取可迁移的方法论片段，标注证据等级。

| 步骤 | 操作 | 产出 |
|------|------|------|
| B1 | 从候选文本中识别方法论片段（教训/模式/规则/反模式） | 片段候选列表 |
| B2 | 为每个片段标注证据等级（A/B/C/D，定义见 §2.3） | 带等级的片段列表 |
| B3 | 描述可迁移条件（适用场景 + 反例/限制） | 含边界条件的片段 |
| B4 | 填写证据卡 §2（关键方法论片段） | 完整证据卡 |
| B5 | 提取者自评：标注不确定项和需要复编码的维度 | 自评注记 |

**片段提取质量标准**（来自 MUST-HAVE 共同元素，Phase 3 对比矩阵）：

每个片段必须包含以下 6 个核心字段：
1. **片段 ID** — 唯一标识（格式：`{项目缩写}-M{序号}`）
2. **描述** — 一句话概括方法论主张（≤50 字）
3. **来源文件 + 行号** — 精确到文件路径 + 行号范围
4. **可迁移条件** — 适用场景 + 前提条件
5. **反例/限制** — 明确什么情况下不适用
6. **证据等级** — A/B/C/D（定义见 §2.3）

以下 2 个发布 Gate 字段：
7. **独立审查状态** — 该片段是否经过独立审查验证
8. **提取者注记** — 提取过程中的不确定项

#### Phase C: 跨项目综合

**目标**：将多个项目的片段横向对比，识别跨项目模式。

| 步骤 | 操作 | 产出 |
|------|------|------|
| C1 | 将 ≥8 个项目的片段按主题聚类 | 主题聚类 |
| C2 | 对每个候选模式执行对抗验证（至少 2 个独立否定者） | 对抗验证结果 |
| C3 | 对不确定项启动 Judge Panel（≥3 个独立判断） | 裁决记录 |
| C4 | 标注模式强度（B/C+/Sp/REJECTED）和源项目数 | 模式目录 |
| C5 | 异后端复编码：关键维度（独有特征、审查状态、源质量）由第二后端独立重编码 | 复编码对比报告 |

**模式强度分级标准**（来自 Phase 4 对抗验证）：

| 强度 | 含义 | 门槛 |
|------|------|------|
| **A** | 强实证模式，可操作化 | ≥3 源项目直接证据 + 对抗验证存活 + 效应量可估计 |
| **A-** | 强过程证据，源项目证据不足 | 核心证据来自本元项目 + 多源项目相关证据 |
| **B** | 有实证锚点，可操作化 | ≥1 源项目直接证据 + 效应量有估计但精度低 |
| **B-** | 有实证锚点，效应量被修正 | ≥1 源项目证据 + 效应量经异后端复验证下调 |
| **C+** | 有限证据，操作化已先行 | 方向性证据存在 + 具体命题被削弱或有反例 |
| **Sp** | 工具能力注记 | N=1 或自评循环 |
| **REJECTED** | 被否决 | 证据被证伪或替代解释更强 |

> **G5 审计注**：本元项目 Phase 4 的 8 个候选模式中，0 个达 A 级，1 个达 A-（P6，核心比较证据来自本元项目），1 个达 B-（P5，效应量经 Qwen 修正），3 个达 C+（P1、P2、P3），1 个标 Sp（P8），2 个被拒绝（P4、P7）。详见 §4。

#### Phase D: 交付与验证

**目标**：产出框架文档 + 独立审查验证。

| 步骤 | 操作 | 产出 |
|------|------|------|
| D1 | 合成框架文档（.md） | 人读框架文档 |
| D2 | 生成机读 JSON 双件 | 机读框架 JSON |
| D3 | 异后端独立审查（按 §5 SOP） | 审查报告 + 修正记录 |
| D3.5 | 审查发现回写——将影响模式/标签的审查发现更新到 §4.2 模式目录和 §3.2 映射表 | 更新后的模式目录 |
| D4 | 更新项目 CLAUDE.md + DEV_LOG.md | 项目状态更新 |
| D5 | 闭合评估（按父框架闭合规范） | 闭合记录 |

### §2.3 证据等级标注规范

本框架使用四层证据等级（源自 Small Scale 分析的 L1-L4 体系，经 Phase 3 对比矩阵确认与其他项目的 A/B/C/D 等价）：

| 等级 | 定义 | 可支撑的主张强度 | 示例 |
|------|------|----------------|------|
| **A** | 可直接验证的实证：源项目有明确记录、可定位到文件行号、经独立审查确认 | 可操作化规则 | ETF-M1 负迁移检验：CLAUDE.md:172 明确记录 + Kimi/GLM 双审确认 |
| **B** | 有记录但未独立验证：源项目有记录但无独立审查确认，或独立审查提出保留 | 方向性建议 | ETF-M4 审查时机：项目经验总结记录但无独立审查专门验证该条 |
| **C** | 从项目过程中推断：模式存在但证据来自过程推断而非项目明确记录 | 假设，需进一步验证 | P3 方法论资产提炼的事后编号确认偏误 |
| **D** | 理论推测/单次观察：仅来自单个项目的单一事件，或纯理论推导 | 仅记录，不做操作化 | 单次发生的工具错误，无复现证据 |

**C/D 边界判定规则**（Phase 7.5 GLM-5 审查 Finding 2 修正；Phase 7.6 二次修正）：

> **前置过滤**：单次工具错误、一次性疏忽等**无方法论含义**的事件不应进入片段系统（既不是 C 也不是 D，直接排除）。D 级保留给"有方法论含义但仅来自单个项目的单一事件或纯理论推导"。

C 级（从项目过程中推断）与 D 级（理论推测/单次观察）的边界判定流程：

1. **来源是否为项目实际执行过程？**
   - 是（来自 Dev Log / 审查报告 / 项目总结中的实际观察）→ 进入第 2 步
   - 否（来自纯理论推导 / 类比 / 外部文献 / 单次工具错误无方法论含义）→ **D 级或不进入片段**
2. **是否可定位到具体文件/行号？**
   - 是 → 进入第 3 步
   - 否 → **C 级**（有过程痕迹但不精确；标注"无法精确定位"）
3. **是否有 ≥2 个独立来源交叉验证？**
   - 是 → **C 级**（可升级为 B 级的候选，需补充独立审查确认）
   - 否（仅 1 个来源）→ **C 级**（标注"单来源"）

**简化记忆**：不进入片段="无方法论含义的一次性事件" / D="我推测（有方法论含义但无执行痕迹）" / C="我观察到（有痕迹）" / B="我观察到且有人核实过" / A="我观察到且有人核实过且证据可精确复现"

**示例**：
- 一次 API 调用失败，有日志行号 → **不进入片段**（无方法论含义）
- 从一次 API 失败中推断"需要重试机制" → **D 级**（有方法论含义但来自单次事件+推断）
- Dev Log 记录了"忘记在 prompt 中加输出格式要求"（可定位）→ **C 级**（操作失误，有过程痕迹，单来源）

**使用规则**：
- 任何标为 `[S]` 稳定组件的条款，其支撑片段的 A+B 级证据占比必须 ≥60%
- A 级证据可独立支撑 `[E]` 试行协议；B 级需 ≥2 条同方向证据
- C/D 级证据不得单独支撑任何操作化建议，仅作为附录记录

## §3 采纳决策模板 `[试行协议]`

> **稳定性**: `[E]` — 三级分流骨架来自 P3（C+，4 源项目但有事後确认偏误），决策矩阵来自 Phase 4 对抗验证实践。采纳/拒绝标准在本元项目执行 1 次。

### §3.1 三级分流：从失败到方法论资产

**核心理念**（来自并购重组 playbook + 形态匹配 ETF + Small Scale + Evolver，4 源项目支撑）：

不是所有失败和教训都值得进入方法论提取流水线。执行三级分流以控制噪音：

```
项目关闭
  │
  ├── LEVEL 1: 执行层失误
  │     单次操作错误、工具误用、临时疏忽
  │     → 归档到项目经验总结，不进入方法论提取
  │
  ├── LEVEL 2: 项目内模式
  │     在本项目内反复出现，但无跨项目证据
  │     → 记录为"待观察"，等待 ≥1 个其他项目验证后升级
  │
  └── LEVEL 3: 系统性缺陷
        根因跨项目可推广，有 ≥2 个项目间接证据
        → 进入方法论提取流水线

注：三级比例（~60%/~25%/~15%）为启发式估计，非统计校准。P3 存在事后确认偏误——百分比来自事后编号而非事前预注册。待 ≥3 个独立项目预注册验证。

**接口**：§3.1 三级分流在 §2.1 Phase 0（项目筛选）阶段先做项目级预分流，在 Phase B（片段提取）后对每个片段做精确分流。LEVEL 2→3 升级协议：每 3 个新项目闭合后，由独立审查者回顾 LEVEL 2 清单，重新评估是否有应升为 LEVEL 3 的条目。这解决了"项目关闭时无法判断跨项目证据"的时间可行性问题。
```

**分流标准**（三级判定矩阵）：

| 判定维度 | LEVEL 1 | LEVEL 2 | LEVEL 3 |
|---------|---------|---------|---------|
| 复现范围 | 单次 | 本项目反复 | 跨项目 |
| 根因深度 | 操作失误 | 流程缺陷 | 认知/架构偏差 |
| 可迁移性 | 无 | 待验证 | 有（含边界条件） |
| 跨项目证据 | 0 | 0（待观察） | ≥2 独立项目证据 |
| 处理方式 | 归档 | 预登记 | 进入提炼流水线 |

> **注**（Phase 7.6 修正）：LEVEL 3 跨项目证据门槛统一为"≥2 个独立项目证据"。仅 1 个间接证据或 2 个推测证据时保留在 LEVEL 2 待观察——待 ≥2 个独立项目证据满足后由独立审查者升级。此前分流矩阵中"≥1 间接或 ≥2 推测"的旧表述已被替换。

**根因深度判定辅助标准**（Phase 7.5 GLM-5 审查 Finding 2 修正）：

| 根因级别 | 判定特征 | 典型示例 | 反例（易误判） |
|---------|---------|---------|-------------|
| **操作失误** | 单次行为错误，换人/重做可避免；与执行者的具体选择直接相关 | "忘记在 prompt 里加输出格式要求"、"用错了 API endpoint" | 不是"反复出现的操作失误"——如果同一操作失误出现 ≥3 次，应升级为流程缺陷（缺乏检查机制） |
| **流程缺陷** | 系统性缺少检查步骤/门禁/验证环节；即使换人也可能重现 | "没有在发布前检查数据来源标注"、"缺少 Phase 级别的独立审查触发机制" | 不是"事后发现有遗漏"——如果遗漏的原因是缺乏标准化的检查清单，根因是流程缺陷而非操作失误 |
| **认知/架构偏差** | 影响多个 Phase/项目的思维模式或架构设计缺陷；换流程也难以消除 | "事后编号导致的确认偏误"、"将模板效应误认为方法论收敛"、"将工作流强制当作方法论发现" | 与 LEVEL 3 门槛一致：需要 ≥2 个独立项目证据才能可靠判定（单项目证据仅达 LEVEL 2） |

**判定流程**：
1. 先判断是否单次 → 是 → LEVEL 1
2. 判断是否缺乏检查/门禁/验证环节 → 是 → LEVEL 2（流程缺陷）
3. 判断是否跨项目反复出现且根因涉及思维模式/架构设计 → 是 + ≥2 项目证据 → LEVEL 3

**举例**：
- P4（证据等级意识结构性出现）被拒绝的根因不是"Phase 4 判断错误"（操作失误），而是"将模板效应误认为方法论收敛"（认知偏差，LEVEL 3）——但因为证据被证伪（6条否决理由），它不进入提炼流水线
- P3（事后确认偏误）的根因是"事后编号而非事前预注册"（认知偏差，LEVEL 3），有 4 个源项目的间接证据

### §3.3 不采纳（反模式）决策模板

**来自 Phase 4 的两个完整拒绝案例**：

#### P4 拒绝案例：证据等级意识结构性出现

**原声称**：A/B/C/D 证据等级在不同项目中自然收敛。

**否决理由**（6 条独立理由）：
1. **事实错误**：GitNexus 并非使用 A/B/C/D 等级
2. **非独立**：Small Scale 的 L1-L4 直系传承自框架 §9.6
3. **同作者 + 72 小时窗口**：所有项目均由同一操作者（用户 + DeepSeek）在 72h 内完成
4. **不同分类维度**：各项目的"等级"分类维度不同（证据强度 vs 验证状态 vs 迁移就绪度）
5. **选择偏差**：仅 3/13 项目有形式化证据体系
6. **替代解释更优**：模板效应（框架 §9.6 提供了统一模板）>> 自然收敛

**教训**：表面收敛 ≠ 真实模式。当多个项目使用相同模板/框架/操作者时，必须排除模板效应。

#### P7 拒绝案例："医生的孩子没人治"

**原声称**：方法论自觉越高的项目，自文档化越弱（讽刺叙事）。

**否决理由**：**关系方向反了**。方法论自觉最高的 3 个项目（并购重组/ETF/PilotDeck）恰好自文档化最强。Agent Harness（方法论自觉最低）自文档化最弱。实际 ρ > 0。

**教训**：讽刺叙事结构吸引力 > 实质时，必须用数据验证方向。

### §3.4 反模式排除清单

以下类型的方法论主张**不进入框架正文**，仅记录在反例附录中：

| 反模式类型 | 特征 | 处理 |
|-----------|------|------|
| 自评循环 | 模式证据完全来自提出者自己的项目 | 标注 Sp，等待外部验证 |
| 单一后端确认 | 仅一个模型后端支持，无复编码 | 标注为"待复编码" |
| 无边界条件 | 声称普遍适用但无反例/限制描述 | 退回补充边界条件 |
| 过度包装 | 将工程 bug 包装为方法论发现 | 降级为工具注记或拒绝 |
| 虚构吸收 | 声称已被某框架/项目吸收但无体内证据 | 拒绝该"已吸收"声称 |

## §4 跨项目模式识别 `[试行协议]`

> **稳定性**: `[E]` — 模式目录来自 Phase 4 对抗验证（17 agents, ~636K tokens），但所有模式强度 ≤B 级，无一达标稳定组件。本目录作为试行模式集，待 ≥3 个独立项目验证后修订。

### §4.1 模式识别流程 `[审查门禁]`

来自本元项目 Phase 4 执行经验（P6 操作化子条款），作为流程 Gate：

1. **候选模式生成**：从 Phase C 主题聚类中提名候选模式（≥2 个源项目片段支撑）
2. **对抗验证**（强制）：每个候选模式分配 ≥2 个独立否定者，prompt 指令为"试图推翻该模式"
3. **不确定项升级**：若否定者间分歧 ≥1 票，启动 Judge Panel（≥3 个独立判断）
4. **异后端复编码**（强制）：独有特征计数、独立审查状态判定、源质量评估三个维度必须由第二后端独立重编码
5. **稳定性标签赋值**：对已有模式（§4.2 目录中），使用预填强度值直接查 §3.2 矩阵赋值；对新候选模式，执行步骤 1-4 后按 §3.2 矩阵赋值。预填值仅在新一轮对抗验证（步骤 2）或独立审查发现影响模式强度后才被更新

**已知局限**：本元项目 Phase 4 Workflow 未记录 agent 级分歧（否定者间的逐条投票），仅记录了最终裁决。这降低了对抗验证过程的可追溯性。后续执行时应强制记录每个否定者的独立投票和理由。

**更新**（Codex 交叉验证后）：本条局限已确认——当前 §4.2 的模式强度判定无法逐条追溯到具体否定者的投票。

**对抗验证 Prompt 模板**（Phase 7.5 GLM-5 审查 Finding 2 修正）：

```
你是一个方法论模式的独立否定者。你的任务不是确认这个模式，而是试图推翻它。

【候选模式】
{模式名称 + 一句话核心命题}

【支撑证据摘要】
{源项目数 + 证据等级分布 + 关键引用}

【推翻策略】
请从以下角度逐一攻击这个模式：

1. **反例攻击**：是否存在与模式声称相反的具体案例？（至少找出 1 个，找不到则说明搜索范围）
2. **替代解释攻击**：支撑证据能否用更简单的解释替代？
   - 模板效应：多个项目的"相似"是否来自共同模板/框架？
   - 选择偏差：是否只选了支持性案例而排除了负面案例？
   - 操作者效应：是否所有证据都来自同一操作者/模型？
   - 共同来源：多个项目是否源自同一上游设计？
3. **逻辑漏洞攻击**：
   - 因果关系方向是否正确？（A→B 还是 B→A？）
   - 是否存在混淆变量？
   - 声称的效应量是否有夸大？
4. **可操作化攻击**：
   - 这个模式能否被转化为具体可执行的规则？
   - 如果不能，它是否只是一个"有趣观察"而非方法论贡献？

【输出格式】
- 投票：推翻 / 保留但削弱 / 保留（三选一）
- 最致命的 1 个攻击点：（≤100 字）
- 如果保留但削弱：建议的边界条件或强度降级
- 如果推翻：推翻的核心理由

【约束】
- 你必须至少提出 1 个实质性攻击点，否则投票无效
- "证据不足"是有效攻击，但必须指明具体缺失什么证据
```

**否定者投票记录规范**（强制）：

每个否定者的输出必须包含以下字段并持久化到审查记录：
```yaml
否定者ID: {agent_id}
后端模型: {model_name}
投票: {推翻 | 保留但削弱 | 保留}
核心攻击点: {≤100字}
建议强度: {A | A- | B | B- | C+ | Sp | REJECTED}
建议边界条件: {如有}
```

≥2 个否定者投票"推翻"→ 模式被拒绝。≥2 个投票"保留但削弱"→ 降级并记录边界条件。票数分散（推翻/保留但削弱/保留各 1）→ 启动 Judge Panel（≥3 个独立判断）。

### §4.2 当前已知模式目录

#### B 级模式（有实证锚点，可操作化）

##### P6: 异后端 Phase 后审查 `[审查门禁]` `强度: A-`

> **强度判定来源与主观性声明**（Phase 7.5 GLM-5 审查 Finding 4 修正）：
> P6 的 A- 级强度来自 Qwen3.7-Max 在 Phase 4 复验证中的升级判定（基于本元项目 4 轮异后端 vs 3 轮同后端的实测比较），非来自源项目直接证据。按 §3.2 采纳矩阵，P6 的「0 源项目核心证据 + 对抗验证存活」对应「审查门禁 [I]」位置，A- 级强度反映本元项目过程证据的强度（本元项目实测效应明确），不代表跨项目可复现强度。
> 
> **两套分类系统的独立变化**（参见 §0.2）：P6 是强度高（A-）但稳定性标签低（[I]）的典型案例——强度衡量证据量（本元项目实测强证据），标签衡量证据来源类型（过程证据非源项目证据）。使用 P6 时应以 [I] 标签为准（审查门禁，不宣称跨项目稳定性），不以 A- 强度为推广依据。

**声称**：在方法论/概念归类项目中，每 Phase 产出后插入一轮异后端独立审查，倾向于在概念落地前发现归类错误。**本元项目观察到**异后端单轮纠偏价值 > 同后端多轮（此为高优先审查门禁假设，非跨源项目已验证规则）。

> 以下为 [I] 级流程规则（基于本元项目执行经验），非方法论主张。格式参考 §4.2 其他模式仅为排版一致性。

**本元项目执行规则**：
- **触发条件**：项目类型 = 方法论/概念归类 + Phase 产出含 ≥3 个概念性判断 + 异后端工具可用
- **执行规则**：审查 prompt 由非作者设计 + 双轴不同（事实轴 + 归类轴）+ 标记修正轮次
- **反例**：纯工程实现项目（代码正确性单轴足够）
- **证据基础**：核心比较来自本元项目（HG-0/Kimi/Codex/Qwen 四轮异后端 vs GPT-5.5 (Codex CLI) 三轮同后端）+ 6 源项目支持"独立审查有价值"

> **G5 注**：P6 是当前操作化最完整的模式，但稳定组件门槛依赖的核心比较证据来自本元项目而非源项目。Phase 5 写入时标注为"本项目实测强证据 + 多源项目相关证据"。

##### P5: 提取者效应 `[试行协议]` `强度: B-（Qwen 复验证修正纯效应 ~9.4%）`

**声称**：单一后端对解释性方法论提取存在系统性倾向。在本元项目观察到的模型对（DeepSeek-V4-Pro vs GPT-5.5 (Codex CLI)）中，DeepSeek 偏向膨胀（独有特征 ~65% inflation），GPT-5.5 偏向保守。

**重要限定**：
1. 证据仅来自 2 份源文档（Small Scale + GitNexus）的 1 对模型复编码（DeepSeek-V4-Pro vs GPT-5.5 (Codex CLI)），不可泛化为模型固有特性
2. Qwen 指出任务规范模糊贡献了相当比例的分歧——提取者效应是"模型 × 任务规范"的交互效应，非模型主效应
3. 风险分级基于当前观察到的模型对 + 当前任务规范；**维度风险非恒定**——更换模型对或标准化 codebook 后应重新评定
4. 将"任务规范标准化程度"作为调节变量：codebook 越标准，提取者效应可能越小

**操作化（风险分级校准）**：
| 维度 | 风险等级 | 复编码要求 |
|------|:------:|------|
| 独有特征计数 | **高** | 强制 ≥2 后端独立计数 |
| 独立审查状态判定 | **高** | 强制 ≥2 后端独立判定 |
| 源质量评估 | **高** | 强制 ≥2 后端独立评分 |
| 片段长度/粒度 | 中 | Spot-check ≥1 份文档 |
| 局限声明详细度 | 中 | Spot-check ≥1 份文档 |
| 维度数/组织方式 | 低 | 豁免（客观可验证） |

- **证据基础**：2 源项目复编码（Small Scale + GitNexus），DeepSeek vs GPT-5.5 (Codex CLI) 显著分歧率 ~22%（Qwen 修正纯提取者效应 ~9.4%）

#### C+ 级模式（有限证据，操作化已先行）

##### P1: Premise Gate `[试行协议]` `强度: C+`

**声称**：若审查 scope 默认为执行正确性且无显式 premise-review gate，premise-level 错误检测延迟显著长于 execution-level 错误。

**操作化**：
- 项目启动前强制回答："这件事该不该做？"（非"这一步做对了吗？"）
- 锁定可证伪死亡判据（Phase 0）
- 独立审查 prompt 必须包含"质疑前提"轴（非仅"检查执行"轴）
- **执行保障**：premise gate 不仅需要设计，还需配套强制执行机制（如独立方在 gate 节点具有否决权，而非仅建议权）

**已知局限**：gate 存在 ≠ gate 有效执行——历史数据无法区分"gate 缺失"和"gate 无效"（ETF 18 版本不能严格证明如果当时有 gate 就一定会提前止损）。

- **证据基础**：2 源项目直接支撑（ETF 18版本教训 + 并购重组 Phase 0 kill-test）
- **待验证**：需要第 3 个独立项目的 premise-gate 缺失 → 补救成本数据

##### P3: 方法论资产提炼协议 `[试行协议]` `强度: C+`

**声称**：项目关闭时对失败/教训执行三级分流。仅 LEVEL 3（系统性缺陷 + 根因跨项目可推广）进入方法论提炼流水线。

**操作化**：
- 关闭前定义分流标准、排除清单、升级条件（预注册）
- 四步提炼：诊断（根因分析）→ 命名（可检索标签）→ 模板化（可操作规则）→ 可迁移（边界条件）
- 事后编号需经独立审查确认非确认偏误

- **证据基础**：4 源项目（ETF 负迁移 + 并购重组闭环规则 + Small Scale P0 防错 + Evolver [Sp] 升级条件）
- **已知偏误**：事后确认偏误（模式编号在事后，非事前预注册）+ 事前归档偏误（分流操作者在项目关闭时可能错误地将跨项目教训归档为项目内模式）。缓解措施：LEVEL 2→3 升级协议（每 3 个新项目闭合后由独立审查者回顾）

##### P2: 多后端审查的多样性保险 `[试行协议]` `强度: C+`

**声称**：多后端可能覆盖更广的错误敏感性空间，但效应大小/方向/最优组合未量化。

**操作化**（修正版，去除已被证伪的"两类错误分化"）：
- 复杂概念/方法论项目应引入认知距离足够的独立审查
- 零卷入 + 多角色 + 多轮是必要条件，多后端是增强项
- 不宣称"多后端一定优于单后端多轮"
- **上游偏差检查**：审查 prompt 本身应由独立方设计（非项目作者），且应包含"质疑问题框架"轴——多个后端回答同一个有偏差的问题是假独立

**已知局限**：多后端不能检测共享上游偏差（所有后端共享同一 prompt 设计/问题框架/评估标准时的系统性盲点）。

- **证据基础**：6 源项目支持"独立审查有价值"，但原"两类错误分化"子命题被 GitNexus 反例证伪

#### 工具注记

##### P8: Dynamic Workflow 并行执行 `[Sp]`

**适用条件**：拓扑独立 + 模板同质 + 可事后批量验证 + N ≥ 3

**注意事项**：同模型并行 agent 产生系统性同质化偏差，须配套异后端审查。

**证据基础**：N=1 效率观察（本元项目），3 源项目记录"使用了 workflow"但无效率测量。

### §4.3 模式演化与更新协议

1. **新项目闭合后**：按 §1 筛选 → 若满足提取条件，按 §2 流程提取 → 将新模式与现有目录对照
2. **模式升级触发**：当某试行协议的源项目直接证据从 1-2 增至 ≥3 且无新反例 → 启动升级评估
3. **模式降级/拒绝触发**：当新项目提供明确反例 → 启动降级或拒绝程序
4. **审查驱动更新**：当独立审查发现影响模式强度判定（证据被修正、反例被新增、操作化前提不成立）→ 在修正框架文档的同时更新 §4.2 模式目录和 §3.2 G5 审计映射表

## §5 独立审查 SOP `[试行协议 + 审查门禁]`

> **稳定性**: `[I]` 审查门禁（流程规则来自本元项目过程证据 + 框架 v1.5.1 §9.2）+ `[E]` 试行协议（审查效果比较声称来自 P6，核心证据在本元项目）。
> **理论基础**: 本 SOP 的多层审查架构可追溯至 **Swiss Cheese Model**（Reason, 1990）——航空安全领域的多层防御理论。每层 QA 屏障有不同类型的"漏洞"（盲审的认知漏洞、答辩的叙事漏洞、复编码的提取者偏倚漏洞），多层不相关屏障叠加后故障穿透概率趋近于零。2026-07-23 的外部验证（CrossCheck 对照，详见 `post-hoc-external-validation.md` §5）确认了此理论框架的适用性，并揭示了被动合规（git hooks）与主动自觉（铁律）之间的互补盲区。

### §5.1 审查触发条件 `[审查门禁]`

以下节点**建议**触发独立审查（[I] 级流程规则，基于本元项目经验；跨项目有效性待独立验证）：

| 触发节点 | 审查对象 | 最低审查强度 | 审查后端要求 |
|---------|---------|:----------:|------------|
| 项目 Spec 完成后 | Spec 完整性 + 前提合理性 | 1 轮 | 异后端 |
| 每个 Phase 产出后 | Phase 产出的事实准确 + 归类正确 | 1 轮 | 异后端 |
| 框架文档初稿后 | 全件审查（事实 + 逻辑 + 完备性） | 1-2 轮 | 异后端 |
| 闭合前 | 闭合条件满足性 | 1 轮 | 异后端 |

**豁免条件**（任一满足即可跳过该节点审查）：
- 该 Phase 产出为纯工程实现（无概念性判断）
- 异后端工具不可用且已记录为已知局限
- 项目为 S-tier 轻量项目（可在闭合前合并为单次终期审查）

### §5.2 审查 Prompt 设计原则 `[试行协议]`

来自本元项目 4 轮审查 + 方法论审查 SOP v1.0.3 的经验提炼：

**双轴结构**（每个审查 prompt 必须包含两条独立轴线）：

| 轴线 | 审查问题 | 适用于 |
|------|---------|--------|
| **事实轴** | "这些声称的事实准确吗？源文件确实说了这些吗？" | 所有 Phase |
| **归类轴** | "这些归类/命名/结论合理吗？有没有替代解释？" | 含概念性判断的 Phase |
| **跨项目推论轴** | "这些片段之间的横向关联和综合推论合理吗？有没有模板效应、选择偏差或共同来源导致的虚假收敛？"（已知盲点：事实正确+归类正确+组合错误的"第三类错误"，如 P4 的伪收敛） | 跨项目综合阶段 |

**Prompt 模板**：

```
你是一个独立方法论审查者。你的任务是审查以下产出，找出其中的错误、遗漏和过度声称。

【审查对象】
{phase_output}

【审查轴1: 事实准确性】
逐条检查以下类型的事实错误：
1. 文件路径/行号引用错误
2. 数据/数字/日期错误
3. 声称来源支持但实际不支持
4. 遗漏关键反例或反面证据

【审查轴2: 归类和结论有效性】
逐条检查以下类型的归类错误：
1. 概念混淆（把 A 当作 B）
2. 过度泛化（从 N=1 推断普遍规律）
3. 确认偏误（选择性引用支持性证据）
4. 替代解释（有没有更简单的解释？）

【审查轴3: 跨项目推论有效性】（跨项目综合阶段专用）
逐条检查以下类型的组合错误：
1. 虚假收敛——多个项目的"相似"特征是否来自共同模板/框架/操作者而非真实收敛？
2. 选择偏差——被纳入模式的项目是否都是支持性案例？有没有被排除的负面案例？
3. 组合错误——片段事实正确 + 归类正确，但跨项目推论错误（如 P4：各项目有证据分级是事实，但"结构性自然出现"是错误推论）
4. 上游偏差——多个项目是否共享同一 prompt 设计/问题框架/评估标准，导致"假独立"？

【输出格式】
对每个发现：
- 严重度: CRITICAL（阻断性）/ MAJOR（显著影响）/ MINOR（边界或建议）
- 类型: 事实错误 / 归类错误 / 遗漏
- 位置: 文件名:行号
- 描述: ≤100字
- 建议修正: ≤100字

【评分】
综合评分: X.X/10
- 事实准确性: X.X/10
- 归类有效性: X.X/10
综合判定: GO（≥7.0）/ GO WITH FIXES（5.5-7.0）/ NO-GO（<5.5）
```

**设计原则**：
1. **非作者设计**：审查 prompt 必须由非产出作者设计（避免审查范围被产出者预设）
2. **反事实强制**：必须明确要求审查者"试图推翻"而非"检查正确性"
3. **具体化输出**：审查输出必须包含文件:行号级别的精确定位
4. **分级响应**：审查结果按严重度（CRITICAL/MAJOR/MINOR）分级，不同级别触发不同响应流程

### §5.3 审查结果分级与响应 `[审查门禁]`

| 审查评分 | 判定 | 响应 |
|:------:|:----:|------|
| ≥7.0/10 | **GO** | 记录审查发现，选择性采纳 MINOR 建议 |
| 5.5-7.0 | **GO WITH FIXES** | 强制修复 CRITICAL + MAJOR；MINOR 选择性采纳；修复后进入下一 Phase |
| <5.5 | **NO-GO** | 不进入下一 Phase；强制修复全部 CRITICAL + MAJOR；重新审查直至 ≥5.5 |

**本元项目实测基准**（供审查预期校准）：

| 审查 | 评分 | 判定 | 强制修复数 | 修订后状态 |
|------|:---:|------|:--------:|------|
| HG-0 Codex (Phase 1) | 7.2 | GO | 5 | 直接通过 |
| Kimi (Phase 2) | 5.9 | GO WITH FIXES | 4 | 修正后通过 |
| Codex (Phase 3) | 6.2 | GO WITH FIXES | 5 | 修正后通过 |
| Qwen (Phase 4) | 5.5 | GO WITH FIXES（下沿） | 4 | 修正中 |

**基准观察**（仅供本元项目参考，不作为外推预期）：本元项目 4 轮审查评分的范围为 5.5-7.2，中位数 6.05。N=1 元项目、4 轮审查，样本不足以推断"同类项目"的预期区间。实际使用时应独立收集审查数据。

### §5.4 复编码门禁 `[审查门禁]`

来自 P5（提取者效应）的操作化要求：

以下三个维度在方法论提取的 Phase C（跨项目综合）阶段**必须**由第二后端独立复编码：

1. **独有特征计数** — 每个项目的"独有特征"数量由第二后端独立计数
2. **独立审查状态判定** — 每个项目的"是否经过独立审查"由第二后端独立核实（区分"分析项目有审查"和"提取文档有审查"）
3. **源项目质量评分** — 每个项目的综合质量评分由第二后端独立评定

**复编码分歧处理**：

| 分歧程度 | 处理 |
|:------:|------|
| 完全一致（分歧率 <10%） | 记录一致性，继续 |
| 轻微分歧（10-20%） | 标注分歧维度，以保守值（更低的计数/评分）为准 |
| 显著分歧（>20%） | 启动第三后端仲裁；分歧维度降级为"不确定" |

**本元项目实测分歧率**：22%（7/32 维度），其中独有特征 inflation ~65%、独立审查状态混淆 1 例。

> **压力测试记录**: `explorations/evidence_card_npgs_external.md`、`explorations/evidence_card_ml_quant_trading.md`

### §10.1 修正 1: Phase 0 合格方法论文档类型扩展

**原定义**（§2.1 Phase 0.1）：源项目需有 ≥1 份"主动方法论文档"（CLAUDE.md / DEV_LOG / project_spec / retrospective）。

**压力测试发现**：ml-quant-trading 无 CLAUDE.md，但 arXiv 论文在方法论提取中起到了类似作用——提供设计意图、理论框架和局限性讨论。有学术论文的项目可提取率为 69%，远高于纯代码项目的 37.5%。

**修正**：Phase 0.1 的判别条件从：

> "源项目是否有 ≥1 份主动方法论文档"

扩展为：

> "源项目是否有 ≥1 份方法论文档（主动: CLAUDE.md / DEV_LOG / project_spec / retrospective；**被动: 学术论文 / 正式技术文档 / 系统性的设计文档**）"

被动方法论文档的提取质量低于主动文档（缺少 AI 协作过程记录、审查链、闭环证据），但可供提取的组件数显著多于无任何文档的项目。

### §10.2 修正 2: 新增 D 级证据等级

**原定义**（§2.3）：A/B/C 三级（A ≥3 项目强证据 / B 级中等 / C 级弱证据 / D 已定义为"单次观察或纯理论推导"）。

**压力测试发现**：外部项目（NPGS、ml-quant-trading）的大量可提取内容属于"仅被动痕迹"——代码结构、commit 历史、目录组织——这些可被任何代码分析工具获取，提取框架在被动痕迹维度上没有增量价值。需要一个新的证据等级来区分"能从代码中直接看到的方法论模式"和"需要方法论自觉才能提取的方法论模式"。

**修正**：细化 D 级定义，增加 D-passive 子级：

| 等级 | 含义 | 示例 |
|------|------|------|
| **C** | 从项目过程中推断（需方法论解读） | 从 PR review 模式推断"审查深度" |
| **D-active** | 单次观察或纯理论推导（原有 D 级） | 从一次 API 失败推断"需要重试机制" |
| **D-passive** | **仅被动代码痕迹，无法区分方法论设计与工程惯例**（新增） | 从目录结构推断"模块分层设计" |

D-passive 级片段**不进入**跨项目模式合成（Phase C），仅作为源项目特征记录。

### §10.3 修正 3: 提取者污染警告

**压力测试发现**：NPGS 的 CLAUDE.md 由 fork 作者而非原项目作者编写。若提取时不区分"原项目的方法论"和"fork 后附加的分析"，会制造虚假的方法论信号。

**修正**：在 §2.1 Phase 0.2（项目筛选）增加判别条件：

> **0.2a 提取者独立性检查（新增）**：若源项目的方法论文档（CLAUDE.md / 分析报告等）由提取者或其关联方编写，该项目的以下维度降低一级证据权重：
> - 方法论自觉评分（从"有"降为"有但非原项目"）
> - 审查记录（从"有审查"降为"仅 fork 后审查"）
> - 闭环修复（从"可观察"降为"仅 fork 后修复"）
>
> 此项检查不影响项目进入提取流程，但所有相关片段必须标注`[提取者污染]`注记。

### §10.4 修正 4: 方法论自觉梯度

**压力测试发现**：可提取率与源项目的"方法论自觉"呈清晰梯度：

| 项目类型 | 方法论自觉 | 可提取率 | 代表 |
|---------|:--------:|:------:|------|
| 自有 AI 协作项目 | ★★★★★ | ~88% | ETF/M&A 等 |
| 学术 OSS（有论文） | ★★★ | ~69% | ml-quant-trading |
| 纯代码 OSS | ★ | ~37.5% | NPGS |

**修正**：在 §0.0（方法论定位）增加局限性声明：

> 提取框架的有效范围局限于有方法论自觉的项目。对无主动方法论文档的项目（约占开源项目的大多数），框架的提取能力骤降至 37-69%，且可提取内容集中于"被动代码痕迹"——这些内容用任何代码分析工具（如 GitNexus）同样可以获取。框架的增量价值主要在"将已有的方法论痕迹系统化"——归纳和命名——而非从零发现。

### §10.5 修正 5: 首个 G5 稳定模式——"对失败诚实"

**背景**：项目最初的 G5 审计结论为"无一模式达到 ≥3 源项目稳定门槛"（详见 §0.1）。此结论在初始集（n=12）中成立。

**扩展集发现**：2026-07-23 将源项目扩展至 19 个后，重新评估发现 **"对失败诚实"**（在 README/文档中诚实记录失败、局限和不成熟，而非包装为成功故事）达到 ≥3 源项目门槛。支持项目：

| 项目 | 证据 |
|------|------|
| 形态匹配 ETF 策略 | README 明确标注"投资上失败"，记录负迁移检验的教训 |
| Prompt-TDD Methodology | 全文核心叙事 = "对照实验失败→修正→再实验"，非成功故事 |
| 本项目（方法论提取方法论） | README 开头即标注"不成熟"、"无一模式达标"，G5 审计结论本身就是对失败的诚实记录 |

**意义**：这是框架产生的**第一个 G5 稳定模式**——不是巧合，而是框架自身逻辑的预期结果：在证据不足时不伪装成熟，本身就满足"对失败诚实"的定义。此模式达标不改变其余 7 个模式的试行状态。

**修正**：在 §4.2（模式目录）中为"对失败诚实"新增条目，标注 `[S]`（首个稳定组件）。

### §10.6 修正 6: Swiss Cheese Model 与 QA 架构理论化

**来源**：2026-07-23 外部压力测试 #3（CrossCheck）的对照分析（详见 `post-hoc-external-validation.md` §5）。

**发现**：本项目的多层 QA 架构（Phase 2 撰前审核 → Phase 5 交叉盲审 → Phase 7 设计回溯 → Phase 8 答辩模拟 → Phase 9 闭环修复）本质上是一种 **Swiss Cheese Model**（Reason, 1990）——每层屏障有不同漏洞类型，多层不相关屏障叠加后故障穿透概率趋近于零。但此前从未被理论化命名。

**修正**：在 §5（独立审查 SOP）开头增加 Swiss Cheese Model 的理论锚定，说明多层审查不是冗余而是瑞士奶酪的各层——每层有不同的"漏洞类型"（盲审 = 认知漏洞，答辩 = 叙事漏洞，复编码 = 提取者偏倚漏洞）。

**与 CrossCheck 的互补性**：CrossCheck 用 git hooks 实现被动合规（机械层），本项目用铁律实现主动独立性（认知层）——两种策略的盲区恰好互补。建议在实际部署中结合两种策略。
