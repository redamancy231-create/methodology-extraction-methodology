## 观察1: 审查链谱系作为纵向时间序列 `[不进入模式目录]`

### 12 源项目的审查模式

多数 12 源项目的独立审查发生在完成后的时间截面上。一个值得注意的例外是 PilotDeck 分析——其 R2 明确读取 R1 报告（`independent_review_round2_report.md:5-7`），R3 读取 R1+R2（`independent_review_round3_report.md:5-7`），分析报告从 v1.2 演进到 v1.3。但即使 PilotDeck 的审查链也是针对同一分析对象的多轮递进，而非跨版本追踪——项目本身的版本（分析报告版本）和框架版本是同一个东西。框架的独特之处在于其**审查跨越框架自身的多个主版本**（v1.3.2→v1.4→v1.5→v1.5.1），且审查者明确基于前序驳回理由检查修正（非仅基于前序发现补充检查）。

### 框架开发史的独特证据

AI 协作框架是唯一一个**审查跨越多个版本、审查者知道前序审查结果**的项目：

| 审查 | 版本 | 后端 | 评分 | 与前序的关系 |
|------|------|------|:---:|------|
| R1 | v1.3.2 | Codex/ChatGPT-5.5 | 4.3/10 驳回 | 初次审查 |
| R2 | v1.4 | Kimi-K2.7-Code | — | 独立审查新版本 |
| R3 | v1.5 | ChatGPT-5.5 | 5.43/10 C+ | 与 R1 同后端不同版本 |
| R4 | v1.5.1 | Codex/ChatGPT-5.5 | 7.2/10 通过 | 明确基于 R1 驳回理由检查修正 |
| R5 | v1.5.1 | Qwen3.7-Max | 6.50/10 B- | 独立审查同版本 |
| — | 冻结期内 | Codex/ChatGPT-5.5 | — | 审查 OPEN-5 关闭 + GLM-5.2 编辑 |

这构成了一个**纵向审查时间序列**——审查者不仅检查产出质量，还检查"前序审查发现的问题是否已被修正"。R4 的审查 prompt 明确包含了 R1 的驳回理由作为检查清单：

> "你是独立零卷入审查者。**这是你第二次审查这个草案**。"（`codex_review_prompt_v151.md:12`）

### 这意味着什么

12 源项目提供了"独立审查有价值"的横向证据（P6 的相关证据）。但框架的审查链提供了 P6 不覆盖的新信息：**审查的价值不仅来自独立性，还来自连续性**——知道"上次审查发现了什么"让后续审查更高效（R4 直接检查 R1 的驳回项而非重新发现）。

这个观察不改变 P6 的强度（P6 的核心比较"异后端 vs 同后端"仍需独立源项目验证），但为未来的审查设计提供了方向性暗示：审查 prompt 应该包含前序审查的发现摘要。

**为何不进入模式目录**：N=1（仅框架自身），且框架正运行 Protocol 3——在试跑完成前，框架自身的审查链不能作为"已验证模式"的证据。

## 观察2: 设计-执行角色分离的涌现结构 `[不进入模式目录]`

### 12 源项目的角色模型

12 个源项目的操作模式是统一的：**同一后端（DeepSeek-V4-Pro）负责设计+执行，异后端仅负责审查**。项目中没有出现"一个后端设计、另一个后端编辑/整合、第三个后端执行"的分工。

### 框架开发史的独特证据

框架 v1.5.1 冻结期出现了计划外的角色分离：

| 角色 | 执行者 | 任务 | 独立性标注 |
|------|--------|------|:--:|
| **作者/设计者** | DeepSeek-V4-Pro (Claude Code) | 框架 v1.0→v1.5.1 设计与写作 | — |
| **编辑者** | GLM-5.2 (ZCode CLI) | 冻结期 11 项编辑（含 OPEN-5 关闭、目录重组、协议清单整合） | `[SEMI-ED]` |
| **编辑审查者** | ChatGPT-5.5 (Codex CLI) | 审查 GLM-5.2 的 11 项编辑 | `[IND]` |
| **冷读 prompt 作者** | GLM-5.2 → Kimi-K2.7-Code | 写冷读 prompt → 六维偏置审查 → 改写 v2 | `[SEMI-ED]` → `[IND]` |
| **协议执行者** | Qwen3.7-Max（待执行） | 冷读框架（不提 OPEN-1 关键词） | `[IND]` |

这个角色分离的基础是框架 §9.2 的显式设计约束（"不要用作者模型自审框架"——`CLAUDE.md:67`），但在冻结期执行中被推到远超出原设计的粒度——原设计只要求"独立审查者 ≠ 作者"，而冻结期实际形成了"编辑者 [SEMI-ED] + 编辑审查者 [IND] + prompt 偏置审查者 [IND] + 协议执行者 [IND]"的四角色结构。其独特之处不在于"角色分离"的原则（原则是设计的），而在于标注的粒度和分离执行的彻底程度。

最值得注意的案例是**冷读 prompt 的偏置审查链**：
1. GLM-5.2 写冷读 prompt v1 → 标注 `[SEMI-ED]`（编辑独立于内容创作，但非独立审查）
2. ChatGPT-5.5 在编辑复核中指出冷读 prompt 需独立复核（`codex_review_glm52_edits_report.md:78-80`）
3. Kimi-K2.7-Code 执行五维偏置审查 → 判定原 prompt C 级偏置（方向性泄漏/严重性诱导/搜索空间收窄/确认偏误放大/作者意识诱导），改写为 v2
4. v2 prompt 经 Kimi 确认消除 5 类偏置 → 交付给 Qwen 执行

这个 chain 的长度（4 步 × 3 后端）和角色标注的粒度（`[SEMI-ED]` vs `[IND]`）在现有源项目中未见同等复杂度。

### 这意味着什么

P5（提取者效应）关注的是"同一后端编码的偏误"。但框架的冻结期揭示了另一类偏误：**prompt 设计偏误**——即使 prompt 设计者不是框架作者（GLM-5.2 ≠ DeepSeek），其 prompt 仍有 C 级方向性偏置。这暗示提取者效应可能不仅存在于"编码"阶段，也存在于"prompt 设计"阶段。

**为何不进入模式目录**：该观察来自框架开发史（1 个项目）。角色分离的原则是 §9.2 显式设计的，但其执行粒度（四角色 × 三后端 × 正式标注）受冻结期约束强化——在其他项目中，同样的 §9.2 约束可能产生不同的角色结构。不可直接推广，但可作为 P5 的"待扩展维度"记录。

## 观察3: 递归自检揭示的框架天然边界 `[直接相关G5审计]`

### 递归的悖论

方法论提取框架声称"从闭合项目中提取可复用方法论模式"并"要求 ≥3 源项目证据才能标为稳定组件"。框架自身的开发史（v1.0→v1.5.1）是第 13 个潜在提取源——但它也是运行 Protocol 3 试跑的**同一个项目**。

这产生了一个悖论：
- 若框架将自身开发史作为源项目提取 → 证据来自"正在试跑 Protocol 3 的项目"，而 Protocol 3 试跑尚未完成（Phase 8 Retrospect 未产出）
- 若框架不将自身开发史作为源项目提取 → 放弃了一个包含独特证据类型（纵向审查链、协议迭代记录、冻结期纪律）的项目

G5 审计解决了这个悖论：**框架自身证据不能计入稳定组件门槛**。但 Phase 6 的三项观察表明，这个约束不是因为框架证据"质量不够"，而是因为**递归自检天然无法满足独立性要求**——工具不能同时是工具的验证者。

### 对框架 v0.1 的元层次含义

| 框架声称 | 递归边界 |
|---------|---------|
| "≥3 源项目直接证据才能标稳定组件" | 框架自身永远不能成为这 3 个源项目之一——无论其开发史多丰富 |
| "试行协议待独立项目验证后升级" | "独立项目"的定义必须排除框架自身——即使框架被另一个项目用作工具 |
| "5 组件全部标试行协议" | 这个"全部"是结构性的，不是暂时的——第 5 个组件的验证天然比前 4 个更难（因为它包含审查框架的规则） |

### 对后续工作的含义

1. **Phase 7 终期审查的自我参照问题**：§5 SOP 是框架自己写的，用自己写的 SOP 审查自己——这本身是 P5 要防止的偏误。Phase 7 的定制 prompt 必须由非 DeepSeek 后端设计（而不只是执行）。
2. **框架升级依赖外部性**：v0.1→v1.0 的升级条件（附录 D）隐含了一个假设——未来会有独立研究者使用本框架提取方法论。这个假设未经验证。
3. **递归边界不是框架缺陷**：它是方法论提取作为一个活动的天然特征。任何"如何提取方法论"的框架在应用于自身时都会遇到同样的边界。记录这个边界本身就是 Phase 6 的信息增量。

**为何不进入模式目录**：这不是一个"模式"，而是框架的元层次属性。它不宣称跨项目可推广——它描述的是递归自检的天然限制。

## Mini-extraction 结论

三项观察均不进入模式目录（P1-P8），不触发任何模式的升级评估，不反向改写 Phase 4 结论。它们记录的是：

1. **审查链纵向时间序列**（→ 暗示审查连续性可能增值，但 N=1）
2. **设计-执行角色分离的涌现**（→ 暗示 prompt 设计偏误是 P5 的未覆盖维度，但 N=1）
3. **递归自检的天然边界**（→ 确认 G5 约束是结构性而非暂时性的）

三项观察共同指向一个方向：**框架开发史包含了 12 源项目不具备的证据类型（纵向、角色分离、递归），但这些证据类型本身就是"元层次"的——它们不能通过框架自身的验证标准（≥3 独立源项目）来验证自身。**这个发现不是框架的失败，而是方法论提取作为活动的递归边界。
