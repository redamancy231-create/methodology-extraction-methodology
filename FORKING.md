# Fork 指南 — 方法论提取方法论

> 生成日期: 2026-07-24 · 生成模型: DeepSeek-V4-Pro (via Claude Code CLI)

如果你 fork 了这个仓库，以下是你可以做的修改方向，按投入成本从低到高排列。

---

## 第一层：直接可用（换数据源即可）

这些不需要改方法论本身，只需要替换源项目数据：

| 方向 | 改什么 | 产出 |
|------|--------|------|
| **用自己的项目重跑提取** | 替换 `explorations/evidence_card_*.md` | 自己的方法论特征报告 |
| **用自己的项目重跑 G5 审计** | 替换源项目后按 `synthesis/phase4_traceability_audit.md` 逻辑重跑 | 自己的模式稳定度评估 |
| **追加新审查后端** | 在 `reviews/` 下新增一轮异后端审查 | 扩展审查链数据集 |
| **更新与父框架的对照** | 本项目基于 AI 协作框架 v1.5.1，父框架已演进，可重做 Protocol 3 试跑 | 新版本的试跑报告 |

## 第二层：改进框架本身（需要方法论判断）

| 方向 | 改什么 | 产出 |
|------|--------|------|
| **修正提取模板** | `framework_output/main/方法论提取框架_v0.1_trial.md` §2.2 | 更适合某类项目的提取模板 |
| **新增组件** | 框架新增第 6-8 个组件 | 覆盖当前框架未触及的方法论维度 |
| **调整证据等级标准** | §0.2 的 [S]/[E]/[I]/[Sp] 标签定义 | 更严格或更宽松的升级门槛 |
| **替换/改进审查 SOP** | §5 独立审查 SOP | 适配不同审查场景或审查后端 |
| **补充反例附录** | 新增提取失败或部分失败的案例 | 框架适用边界的负样本 |

## 第三层：转化形式（性质改变）

| 方向 | 改什么 | 产出 |
|------|--------|------|
| **做成 CLI 工具** | 把提取协议写成可执行脚本 | 类似 [CrossCheck](https://github.com/sburl/CrossCheck) 的自动化工具 |
| **做成 GitHub Action** | 提取流程 → CI 模板 | 每次 commit 自动评估方法论文档完整度 |
| **写成学术论文** | 重组实证数据为论文格式 | 可投稿的论文 |
| **做成课程材料** | 把 10 轮审查链变成教学案例 | AI 协作方法论课程 |
| **翻译** | 中→英/日/其他语言 | 扩大读者群 |
| **做成模板仓库** | 精简为 GitHub template repo | 其他人一键生成自己的方法论提取项目 |

## 第四层：扩样本（最需要外部贡献）

这是本项目当前最大的瓶颈——16 个自有源项目来自同一作者，3 个外部项目也由同一操作者执行提取/审查。提取出的模式无法区分"通用方法论"和"个人工作习惯"。需要不同作者的项目（且由不同操作者提取）来验证泛化性。

| 方向 | 改什么 | 产出 |
|------|--------|------|
| **贡献外部项目证据卡** | 新增 `explorations/evidence_card_*.md` | 打破 n=同作者的局限 |
| **贡献外部审查报告** | 新增 `reviews/` 条目 | 扩充审查链数据集 |
| **贡献反例** | 提取失败的项目记录 | 框架适用边界的负样本 |
| **用不同后端重跑全部审查** | 替换审查后端组合 | 多后端独立性是否跨审查者群体成立 |

## 已知局限（fork 前请读）

1. **框架本身不成熟**：v0.1 trial，全部 5 组件均未达到 ≥3 源稳定门槛
2. **同操作者样本**：16 个自有项目来自同一作者，3 个外部项目也由同一操作者执行提取/审查，泛化性未验证
3. **可提取率与项目类型强相关**：自有工具类 ~100% → 纯代码 OSS ~37.5%，详见 `explorations/source_projects_summary.md`
4. **依赖方法论文档的存在**：如果源项目没有 CLAUDE.md/DEV_LOG/审查报告等方法论文档，提取框架几乎无法运作
5. **中文为主**：大部分内容为中文，英文读者需翻译
