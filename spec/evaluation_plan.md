## 三、证据治理规范（Codex HG-0审查 M1-M3 落地）

### 3.1 最小证据包协议（每个源项目）

CLAUDE.md 作为入口，但每个项目至少补读以下文件：
1. README / project spec / 项目主文档
2. Dev Log / Retrospect / 决策记录（如存在）
3. 已有方法论提取文档（如存在）
4. 1-3个关键产物或关键失败记录（按项目类型选择）

### 3.2 证据卡模板

每个源项目产出统一证据卡：

```yaml
项目名称: 
项目类型: [量化/学术/AI分析/元项目/其他]
状态: [闭合/归档/搁置/活跃]
规模: [大(>100文件)/中(10-100文件)/小(<10文件)]
证据来源:
  - 类型: [原始日志/项目总结/审查者推断]
    文件: 
    覆盖度: [完整/抽样/仅摘要]
关键方法论片段:
  - 片段ID:
    描述:
    可迁移条件:
    反例/限制:
已有提取质量: [独立提取文档/嵌入式/无]
置信度: [A:原始记录支持/B:项目总结支持/C:审查者归纳/D:推测]
```

### 3.3 证据等级定义

| 等级 | 含义 | 可使用范围 |
|------|------|-----------|
| **A** | 原始过程记录支持（Dev Log/审查报告/闭合记录） | 框架正文 |
| **B** | 项目总结支持（CLAUDE.md/终期报告/Retrospect） | 框架正文 |
| **C** | 审查者归纳（跨项目推断，非源项目直接记录） | 框架正文，标注[C] |
| **D** | 推测 | 仅待验证区，不进框架正文 |

### 3.4 活跃项目隔离

BDC2026 为活跃项目（非闭合），处理规则：
- 证据卡单独标注 `状态: 活跃`
- 归纳权重降级（不作为跨项目模式的主证据源）
- 定位为"迁移验证案例"——验证已闭合项目的模式是否在活跃项目中可见

### 3.5 Phase 4 进入条件（闸门）

以下条件全部满足方进入 Phase 4：
1. ≥8/12 项目证据卡完成
2. 证据卡覆盖大/中/小项目类型
3. 已有提取产物格式差异已完成中间表示映射（不可直接综合的，先做中间表示）
