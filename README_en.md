## What This Is

A meta-level methodology project that systematically extracts reusable methodology patterns from **22 source projects**. Source projects include the author's own projects (7 public, 3 private) and independent analyses of other people's open-source projects or papers (12). See `explorations/source_projects_summary.md` for the full list and classification.

It has a **dual purpose**:
1. **Output layer**: Attempt to formalize a "methodology extraction framework"—how to identify, validate, and classify cross-project methodology patterns from a set of projects
2. **Testing layer**: Serve as the first end-to-end trial run of Protocol 3 in the [AI Collaboration Full-Lifecycle Framework](https://github.com/redamancy231-create/ai-collaboration-framework)

## Why It Is Immature

The project's own G5 Traceability Audit concluded: **No methodology pattern meets the stability threshold for components, which requires support from ≥3 source projects.** The 22 source projects consist of **12 independent analyses of other people's open-source projects or papers** (Evolver, GitNexus, PilotDeck, Small Scale, Agent Harness, LIT, claude-code-ultimate-guide, claude-md-generation, PocketFlow, NPGS, ml-quant-trading, CrossCheck), **7 author-built public projects** (AI Collaboration Framework, claude-skills, docx-pipeline, ETF-pybind11, Independent Review Toolkit, M&A Case Study, prompt-tdd), and 3 author-built private projects. All extraction and review work was performed by the same operator. This sample is too small to support generalizable methodology extraction.

In other words, we proved "how difficult this is," but we have not yet established "how this should be done."

## What Is Worth Reading

### Empirical Evidence from the Review Chain (The Most Valuable Part)

Ten formal review rounds were conducted using four review backends—GPT-5.5, Kimi-K2.7, Qwen3.7-Max, and GLM-5—plus one re-encoding by a different backend in Phase 3.5. The process produced 58 findings, all of which were closed. Based on the independence analysis in Kohli (2026-05), these represent approximately 2–3 effectively independent perspectives. Key data points:

- **Phase 5 dual-round orthogonal review**: GPT-5.5 found 17 issues and Qwen3.7-Max found 16, with **zero overlap**. No overlap was observed across these two review rounds under the current coding rules, suggesting complementary error sensitivity across models
- **Phase 7.6 final review**: GPT-5.5 rated practical completeness at 7.7/10
- **Methodological value of the reviews**: The full review chain is itself strong evidence for the effectiveness of independent multi-model review

See [`reviews/`](reviews/) for details.

This project was not conceived in advance. The idea emerged in conversation on 2026-06-16 during the Protocol 3 trial run of the [AI Collaboration Full-Lifecycle Framework](https://github.com/redamancy231-create/ai-collaboration-framework), proposed by an AI model (DeepSeek-V4-Pro or another collaborating model used during the same period). Each constituent element of the idea has precedents in academic traditions: **systematic literature review methodology** (standard methods in medicine and the social sciences), **lessons-learned extraction in software engineering** (lessons learned / post-mortem analysis), and **evidence grading systems** (the [S]/[E]/[I] labels were designed within the framework itself). ### Academic Literature

The project's central claim—the value of independent multi-model review—received external support and points of comparison in the 2026 academic literature:

**「九法官，两有效票」 ("Nine Judges, Two Effective Votes") (Kohli, 2026-05)** — [*Nine Judges, Two Effective Votes: Correlated Errors Undermine LLM Evaluation Panels*](https://arxiv.org/abs/2605.29800). Core finding: a panel of nine frontier LLMs yielded only ~2 effectively independent votes; **the single best judge matched or outperformed the entire panel**. This provides an external reference point for this project's multi-model review framework. On the classification tasks tested by Kohli, four to five models may already represent the effective upper limit, with diminishing returns from adding more models, though extrapolation across tasks requires caution.

**「行为纠缠」 ("Behavioral Entanglement") statistical framework (Kuai et al., 2026-04)** — [*How Independent are Large Language Models? A Statistical Framework for Auditing Behavioral Entanglement*](https://arxiv.org/abs/2604.07650). Testing 18 LLMs from six model families, the study found widespread "behavioral entanglement," meaning implicit dependence caused by shared training data, distillation, or alignment pipelines. An ensemble of disentangled verifiers improved accuracy by 4.5%.

**「当模型意见分歧时」 ("When Models Disagree") (Nájera et al., 2026-05)** — [*When Models Disagree: Rethinking LLM Evaluation for Public Comment Analysis*](https://arxiv.org/abs/2605.29025). This work reframes multi-model disagreement as a **diagnostic signal** rather than a defect: disagreement patterns map onto genuine interpretive complexity. The 33 findings from this project's two Phase 5 review rounds had no observed overlap under the current coding rules, offering a possible data point for this theory, with the caveat that the review task differs from the paper's public-comment analysis setting.

**「人主导的 AI 共创」 ("Human-Led AI Co-Creation") (Oda, 2026)** — [*Human-Led AI Co-Creation: A Practical Framework for Multi-Model Research, Convergence, and Human Authority*](https://www.academia.edu/165538962) (an independent white paper with unknown peer-review status). Its four-role framework—primary AI drafts → multiple models conduct parallel reviews → iterative cross-model comparison → human makes the final decision—is nearly isomorphic to this project's pipeline.

**Key comparison**:

| Dimension | Academic frontier | This project |
|-----------|-------------------|--------------|
| Quantifying multi-model independence | 9→2 effective votes (Kohli 2026-05) | Complementary error spaces across five models (empirical evidence, without mathematical modeling) |
| Disagreement as a diagnostic signal | Four-category disagreement taxonomy (Nájera 2026-05) | Zero overlap in the two Phase 5 review rounds (empirical case) |
| Human–AI collaboration framework | Four-role model (Oda 2026) | Five iron rules + role slots (stricter independence constraints) |
| **Cross-project pattern extraction** | **No comparable work retrieved** | **Systematic extraction from 12 projects (search has known limitations; see above)** |

### Post-Closure External Validation Appendix

Using the analytical frameworks from the four papers above, we reanalyzed this project's review data by reclassifying all 58 findings, assessing model independence, and comparing collaboration frameworks. See [`post-hoc-external-validation.md`](post-hoc-external-validation.md) for details.

### Four Revisions from External Validation

| Source | What It Validated | What It Revised |
|--------|-------------------|-----------------|
| Kohli (2026-05) | A four-to-five-model framework is the optimal configuration | Ten review rounds ≈ 2–3 effectively independent perspectives; we should not claim "10 independent rounds" |
| Nájera (2026-05) | Zero overlap in Phase 5 is an expected manifestation of "deep disagreement" | The 58 findings can be systematically classified into four categories, with 34% reflecting deep disagreement—making ≥3 backends necessary |
| Kuai (2026-04) | Selecting models from entirely different providers is an implicit disentanglement strategy | Entanglement is low but nonzero; we cannot claim "complete independence" |
| Oda (2026) | The iron-rules system is stricter than the closest academic framework | Oda's failure-mode taxonomy can be incorporated into the Phase 0 risk register |

### Positioning Summary

The project's central claim—that independent multi-model review is effective—received external support and points of comparison from the 2026 academic literature. However, the meta-level action of "systematically extracting reusable methodology literature from multiple AI collaboration projects" had **no comparable work retrieved** in a systematic search of GitHub across nine directions and of the academic literature conducted on 2026-07-23. The search has known limitations, including incomplete GitHub indexing of micro-repositories and the sensitivity of academic searches to keyword selection, so relevant work may exist without having been found. This project is limited to an n=12 same-author sample, and the extracted patterns have not yet reached the stability threshold for cross-user generalization.

### Intended Audience and Value

**Intended audience**: People who already use AI across multiple projects and have accumulated substantial numbers of CLAUDE.md files, DEV_LOG files, and review reports, but feel that their methodology is scattered across those projects and want to systematize it.

**Not intended for**: People who have completed only one or two projects (n is too small); pure coding projects that do not involve AI collaboration; projects without methodology documentation; or people seeking a "best-practices template" (this framework tells you "what you did," not "what you should do").

**Three forms of value**:

1. **Determine what your methodology is "worth"**—the framework's eight components form a checklist. Running it once shows which dimensions of your projects demonstrate methodological awareness—documentation, review, and closure—and which remain blank
2. **Assemble scattered fragments into patterns**—methodology documentation from 20 projects is distributed across many locations; the extraction framework provides a process for cross-project comparison
3. **Enable external validation of your methodology**—once organized into a structured form, it can be compared, challenged, and cross-validated against the academic literature

**Core limitation**: The framework has been run on 22 source projects — 12 externally-analyzed + 7 author-built public + 3 author-built private. Of these, 11 have standalone evidence cards (`explorations/`); the analysis process for the remaining 11 is documented in the review reports under `reviews/`. Extractability correlates with a project's methodological awareness: projects that actively document their AI collaboration process yield most methodology components; projects that leave only passive code traces yield substantially fewer components—the framework cannot extract information that does not exist. The same operator performed all extraction and review work. See `explorations/source_projects_summary.md` for the full list and classification. Whether the framework's effectiveness generalizes to other authors' working habits and other operators' extraction practices **has not been validated**.
