# Phase 7.6 针对性重审报告 — 实用完整性

## 0. 独立性声明

- **后端身份**：Codex / GPT-5（Codex CLI 环境）。
- **前序参与情况**：本次执行实例未参与本项目此前产出或审查；但同类 Codex 后端曾在 Phase 5 中作为 R1 参与。因此本报告按当前主文档重新取证，不沿用旧版印象或旧审查结论。
- **审查角度差异**：Phase 5 侧重框架完整性与交叉审查，Phase 7 侧重闭合/合规，Phase 7.5 侧重原创性与迁移价值；本次仅评估当前文档能否被未参与项目的独立研究者照着实际使用，重点看首次 2 小时路径、操作标准和自限性提示。

## 1. 综合结论

- 综合评分：7.7/10
- 首次使用体验：7.4/10
- 操作化充分性：7.1/10
- 诚实度与自限性：9.0/10
- 判定：GO

当前版本已经从“项目报告”明显推进到“可试用操作手册”：定位、适用范围、前提条件、证据卡、对抗验证 prompt、审查 prompt 都足够让新用户启动一次 2 小时试点。主要短板不是宏观结构，而是若干执行接口仍会让两个新用户在同一步做出不同判断，尤其是 §2.2 模板 rubric 与 §1.2 rubric 不一致、§2.3 C/D 边界规则自相矛盾、§3.1 LEVEL 3 证据门槛有两套表述。

## 2. 三维详细评估

### 2.1 首次使用体验

新用户能在 5 分钟内理解文档用途。标题、版本栏、§0.0 明确说明这是“方法论提取实践的系统化总结”而非成熟方法论创新；§0.1 直接给出 G5 审计结论：“当前无一模式满足 ≥3 源项目直接证据”。这降低了误读为成熟框架的风险。

适用性判断总体清楚。§0.4 给出适用/不适用项目类型、源项目选择偏差表、4 项适用性自检，并提供“≥2 项谨慎则先走 2 小时试点”的决策规则。前提条件也清楚：§0.5 列出闭合记录、独立审查报告、多后端、文件完整性及替代路径。

2 小时路径可启动，但不是完全无摩擦。§1.1 准入 checklist、§2.1 Phase A-B、§2.2 证据卡模板构成了可执行路径；§1.2 的“可迁移片段”三项计数规则也帮助 B1 识别片段。最可能卡住的位置有两处：第一，§2.2 证据卡中的 `rubric评分` 字段使用 D1_结构完整性、D2_证据等级标注、D4_可迁移性等字段，但 §1.2 的正式评分维度是方法论密度、失败教训、独立审查深度、格式成熟度、源项目质量，二者无法直接映射。第二，文档没有一张填好的 2 小时路径示例卡；新用户知道要填什么字段，但不一定知道一条“合格片段”的粒度和写法。

卡住后的求助方向部分存在：缺失证据时 §0.5 和 §2.1 提供降级/标注方案，证据等级可查 §2.3，术语可查附录 A。但对于“我这个片段到底算不算方法论片段”“rubric 字段该按哪张表填”这类首次使用问题，文档目前没有故障排查小节或示例。

### 2.2 操作化充分性

§1.2 可迁移片段计数规则比旧式主观判断强很多：关键词、来源定位、边界条件三项均为可检查条件。剩余歧义在“文本段落”的切分单位：Markdown bullet、表格行、长段落拆分时，不同用户可能计数不同。例如一个表格行同时包含 3 条规则，用户 A 可能计 1 个片段，用户 B 可能拆为 3 个片段。

§3.1 根因深度判定的辅助标准覆盖了常见歧义，尤其是“反复操作失误应升级为流程缺陷”“模板效应误判为方法论收敛是认知偏差”等反例很有用。但 LEVEL 3 门槛存在不一致：三级矩阵写的是“≥1 间接或 ≥2 推测”，而认知/架构偏差辅助标准和判定流程又要求“≥2 个独立项目证据”。在只有 1 个间接证据 + 1 个强推测证据时，不同用户可能分别判为 LEVEL 3 或 LEVEL 2。

§2.3 C/D 边界三步流程没有完全消除模糊地带，因为它与 D 级定义冲突。定义中 D 包含“单个项目的单一事件”，但判定流程中只要来源是项目实际执行过程，即使仅 1 个来源，也被标为 C。于是“某项目一次工具错误，有日志行号，但无复现”按定义是 D，按流程是 C。这个冲突会直接影响片段能否进入后续操作化，因为 C/D 均不能单独支撑操作建议，但 C 会被看作“有痕迹的假设”，D 只是记录。

§4.1 对抗验证已经足够直接使用：有 prompt、攻击角度、输出格式、投票记录 YAML、投票结果处理规则。小缺口是“独立否定者”的独立性标准未具体化，例如同一后端不同会话、同一 prompt 的多 agent、不同后端但共享上游提示词，是否算独立仍需用户自行判断。

§5.2 审查 prompt 模板可以直接复制粘贴使用，事实轴、归类轴、跨项目推论轴覆盖本项目暴露出的主要盲点。小问题是输出类型只给“事实错误 / 归类错误 / 遗漏”，没有显式包含“跨项目推论错误 / 过度声称”，审查者可能把第三类错误硬塞进归类错误。

### 2.3 诚实度与自限性

这是当前文档最强的维度。文档在标题、版本栏、§0.0、§0.1、§0.2、各大章节稳定性说明和末尾注记中反复声明“全组件试行版”“无一组件标 [S]”“未经独立新项目复现”。新用户不太可能误以为这是成熟框架。

源项目选择偏差写得足够具体：项目类型同质、操作者单一、规模偏倚、闭合状态偏倚、AI 模型偏倚均被列出，并说明对可迁移性的影响。§0.4 的适用性自检也把偏差转化成了使用前判断。

P6 的 A- 与 [I] 区分解释清晰。§0.2 先解释强度与稳定性标签是两套系统，§3.2 映射表把 P6 放在“0 核心源项目证据 / 审查门禁 [I]”，§4.2 又单独说明 A- 仅反映本元项目过程证据强度，不代表跨项目可复现强度。这是足够克制的写法。

已知局限并非只在开头出现，而是分布在 §2 流程稳定性、§3 分流偏误、§4 模式目录、§5 审查 SOP、P1/P2/P3/P5/P6/P8 各模式说明中。诚实度不构成扣分原因。

## 3. 发现清单

### MAJOR

1. **§2.2 证据卡 rubric 与 §1.2 正式评分 rubric 不一致，影响 2 小时路径填表。**  
   §1.2 的评分维度是方法论密度、失败教训、独立审查深度、格式成熟度、源项目质量；§2.2 模板却要求 D1_结构完整性、D2_证据等级标注、D4_可迁移性等字段。新用户到“填写证据卡”步骤会不知道按哪套标准打分。

2. **§2.3 C/D 边界规则内部冲突。**  
   D 级定义包含“单个项目的单一事件”，但三步判定流程把来自项目实际执行过程且只有 1 个来源的观察判为 C。两个用户面对“有日志的一次性工具错误”时会分别给 C 或 D。

3. **§3.1 LEVEL 3 跨项目证据门槛有两套标准。**  
   分流矩阵允许“≥1 间接或 ≥2 推测”，但认知/架构偏差判定要求“≥2 个独立项目证据”。在边界案例中，这会改变片段是否进入方法论提炼流水线。

4. **缺少一张完整的 2 小时路径示例卡。**  
   文档提供了步骤和模板，但没有展示从项目文件到 3 个方法论片段的实际填表样例。对未参与本项目的新用户来说，片段粒度、边界条件写法、证据等级标注的实际成品仍需猜。

### MINOR

1. **“文本段落”计数单位未定义。**  
   §1.2 规定一个文本段落计 1 个片段，但未说明 bullet、表格行、长段落拆分如何处理。

2. **§4.1 未定义否定者独立性。**  
   已要求 ≥2 个独立否定者，但未说明同后端不同会话、同 prompt 多 agent、不同后端共享 prompt 是否合格。

3. **§5.2 审查输出类型少一个“跨项目推论错误”。**  
   Prompt 有跨项目推论轴，但输出类型只有事实错误、归类错误、遗漏。

4. **2 小时路径缺少时间盒分配。**  
   §0.5 给出路径和产出，但未说明 §1.1、Phase A、Phase B、填卡各自建议用时，首次用户容易在证据收集阶段超时。

## 4. 与既有审查的增量

本审查的增量不是再判断“是否原创”“是否稳定”“是否合规闭合”，而是发现当前操作化修正后的几个用户执行接口问题：

- 前序修正确实补强了操作标准，但新增标准之间存在局部接口不一致，尤其是 §1.2 与 §2.2 rubric、§2.3 定义与流程、§3.1 矩阵与辅助标准。
- 当前文档已经能说明“为什么谨慎使用”，但对“首次用户照着填出的成品应该长什么样”支持不足。
- Prompt 模板已经可复制，但模板周边的执行环境标准仍不完整，例如否定者独立性、审查发现类型枚举。

这些问题不会推翻当前 GO 判定，但会影响独立研究者第一次试跑的一致性。

## 5. 改进建议

评分 ≥7.0，建议在 Phase 8 优先做以下优化：

1. **统一 §1.2 与 §2.2 rubric。**  
   二选一：要么把 §2.2 模板字段改为 §1.2 的五个评分维度；要么在模板中增加“字段映射表”，说明 D1-D5 如何对应 §1.2。

2. **修正 C/D 边界。**  
   明确“单次项目事件”到底是 D，还是“有项目执行痕迹但单来源”的 C。推荐规则：有项目执行痕迹 + 可定位 = C；单次工具错误如无方法论含义则不进入片段，而不是用 D 解决。

3. **统一 LEVEL 3 门槛。**  
   建议采用更保守表述：“LEVEL 3 需 ≥2 个独立项目证据；仅 1 个间接证据或推测证据时保留在 LEVEL 2 待观察。”

4. **新增附录 E：2 小时路径完整样例。**  
   用一个低风险示例项目展示准入检查、Phase A 证据来源表、3 个片段、证据等级、边界条件和提取者注记。这个样例会显著提高独立新用户的一致性。

5. **补充执行细则。**  
   定义段落计数单位、否定者独立性标准、2 小时时间盒、以及 §5.2 输出类型中的“跨项目推论错误”。
